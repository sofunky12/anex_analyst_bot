# Telegram Chat Analytics

Telegram-бот, который собирает активность чата в SQLite (кто, когда и что писал,
счётчики реакций) с шифрованием текста сообщений, плюс Streamlit-дашборд поверх
этих данных. Постоянный сбор идёт через Bot API (`bot.py`) — чат подключается и
отключается динамически командами прямо в самом чате. Личный аккаунт (Telethon)
нужен только разово — догрузить историю чата с момента добавления бота назад.

## Структура проекта

- `src/bot.py` — постоянный сбор через Bot API (aiogram), команды `/activate`,
  `/deactivate`, `/status`, `/chats`, `/import_history`
- `src/history.py` — точечная догрузка истории чата личным аккаунтом (Telethon),
  запускается только на время команды `/import_history`
- `src/crypto.py` — шифрование/расшифровка текста сообщений
- `src/generate_key.py` — генератор ключа шифрования (разовая утилита)
- `src/db.py` — схема SQLite и все аналитические SQL-запросы
- `src/dashboard.py` — Streamlit-дашборд, только читает `activity.db`
- `data/activity.db` — база SQLite, создаётся автоматически при первом запуске
  `src/bot.py`
- `deploy/Dockerfile`, `deploy/fly.toml`, `deploy/start.sh` — деплой на Fly.io
  (фаза 2), см. «Деплой на Fly.io» ниже

## Установка

```bash
pip install -r requirements.txt
cp .env.example .env
```

Заполни `.env`:

- `BOT_TOKEN` — токен от [@BotFather](https://t.me/BotFather)
- `OWNER_USER_ID` — твой **числовой** Telegram user id (не путать с `TG_API_ID` —
  это разные вещи). Узнать, например, у `@userinfobot`. Даёт доступ к `/chats` и
  `/import_history` в личке с ботом
- `TG_API_ID` / `TG_API_HASH` — с [my.telegram.org](https://my.telegram.org),
  нужны только для догрузки истории
- `TG_PHONE` — номер телефона личного аккаунта, от лица которого идёт догрузка
- `DB_ENCRYPTION_KEY` — см. раздел «Шифрование» ниже, без этого `bot.py`
  откажется запускаться

## Настройка бота в Telegram

1. Создай бота через [@BotFather](https://t.me/BotFather) (`/newbot`), токен — в
   `BOT_TOKEN`.
2. **Обязательно:** `/setprivacy` → **Disable** для этого бота в BotFather. По
   умолчанию Bot API отдаёт боту только команды и упоминания — обычные сообщения
   чата не приходят вообще.
3. Если бот уже был добавлен в чат до отключения Privacy Mode — **удали его и
   добавь заново**: изменение privacy применяется только при повторном входе в
   чат. Симптом, если забыть это сделать: бот работает, ошибок нет, но в базе
   почти пусто.

## Авторизация личного аккаунта (Telethon)

Нужна один раз — для команды `/import_history`. Сессия логинится заранее в
консоли, `history.py` сам логин не делает (это фоновая часть процесса бота, там
спрашивать код в чате и небезопасно, и бессмысленно — Telegram блокирует коды,
пересланные в самом Telegram):

```bash
python3 -c "
from telethon.sync import TelegramClient
from dotenv import load_dotenv
import os
load_dotenv()
TelegramClient(
    os.getenv('TG_SESSION', 'history_session'),
    int(os.getenv('TG_API_ID')),
    os.getenv('TG_API_HASH'),
).start(phone=os.getenv('TG_PHONE'))
"
```

Спросит код подтверждения (и пароль 2FA, если он включён) прямо в консоли.
Session-файл (`<TG_SESSION>.session`) сохранится рядом с проектом — повторно
логиниться не придётся, пока файл не будет удалён или Telegram не потребует
ре-авторизацию.

## Шифрование

Текст сообщений (поле `text_encrypted` в БД) шифруется всегда — `bot.py`
физически не запустится без настроенного ключа. Остальные поля (кто, когда, тип
сообщения, счётчик реакций) остаются в открытом виде: без них не построить ни
один график, а сами по себе они не раскрывают содержание переписки.

**`bot.py` поддерживает только статический ключ** (`DB_ENCRYPTION_KEY`) — это
долгоживущий процесс без stdin, спросить пароль через `getpass` там негде.

```bash
python src/generate_key.py
```

Результат — в `.env` как `DB_ENCRYPTION_KEY`. Ключ лежит в файле рядом с
остальными секретами: тот, у кого есть доступ к `.env`, может расшифровать всё.

**Парольный режим (`DB_ENCRYPTION_SALT`)** доступен только в `dashboard.py` —
там пароль вводится в сайдбаре при каждом запуске, сам ключ нигде не
сохраняется. Для `bot.py` не работает, см. выше.

## Запуск

Все команды — из корня репозитория (не `cd src`), чтобы `.env` в корне
находился стандартной загрузкой переменных окружения.

```bash
python src/bot.py
```

Команды бота:

| Команда | Где | Кто | Действие |
|---|---|---|---|
| `/activate` | в группе | админ чата или владелец бота | включает сбор в этом чате, публикует объявление об этом в чат |
| `/deactivate` | в группе | админ чата или владелец бота | останавливает сбор; уже собранные данные не трогает |
| `/status` | в группе | любой участник | показывает, идёт ли сейчас сбор |
| `/chats` | в личке с ботом | только `OWNER_USER_ID` | список известных чатов со статусом (🟢 активен / ⚪️ нет) |
| `/import_history <chat_id>` | в личке с ботом | только `OWNER_USER_ID` | догружает историю чата личным аккаунтом в фоне, репортит прогресс |

Сбор в чате начинается только после явного `/activate` — до этого бот получает
апдейты (он состоит в чате), но ничего не пишет в базу. `/import_history`
работает независимо от статуса активации: можно догрузить историю и в
деактивированном чате.

Бота нужно добавить в группу заранее — обычным способом, через «Добавить
участника».

Остановить — `Ctrl+C`. Безопасно перезапускать в любой момент: `INSERT OR
IGNORE` по `(chat_id, message_id)` не даёт дублей, даже если окна live-сбора и
`/import_history` пересекаются.

## Дашборд

```bash
streamlit run src/dashboard.py
```

Откроется браузер на `localhost:8501`. Можно держать запущенным одновременно с
`bot.py` — они читают/пишут одну базу через WAL-режим SQLite, не мешая друг
другу.

**Сайдбар слева** — выбор чата (если их несколько), период (последние 7/30/90
дней, всё время или свой диапазон), переключатель «только текстовые сообщения»
и, если используется парольный режим шифрования, поле для ввода пароля.

**Вкладки:**
- **Обзор** — сводка, тепловая карта активности (день недели × час), тренд по дням,
  графики по дням недели и по часам
- **Участники** — топ активных участников, таблица и график
- **Сообщения** — разбивка по типам сообщений со спарклайном тренда для каждого
  типа, топ сообщений по реакциям
- **Текст** — самая частая буква и частотный график по буквам

Без ключа шифрования (или до ввода пароля в сайдбаре) вкладка «Текст» и колонка
«Сообщение» в топе по реакциям показывают 🔒 вместо содержимого — всё остальное
(кто, когда, сколько, какого типа) работает как обычно, потому что не требует
расшифровки.

Данные кешируются на 30 секунд — кнопка «Обновить данные» в сайдбаре сбрасывает кеш
сразу.

## Деплой на Fly.io

Фаза 2: `bot.py` и `dashboard.py` работают на Fly.io **в одной Fly Machine**
(`deploy/fly.toml`, `deploy/Dockerfile`, `deploy/start.sh`) — запускаются
вместе через обёрточный скрипт, с общим persistent volume под `activity.db`
и Telethon session и секретами вместо `.env`. Не два отдельных
`[processes]`/машины — volume на Fly физически привязывается только к одной
машине, так что общий диск возможен только внутри одной. Подробности и
история решения — в `CLAUDE.md`, раздел «Хостинг — Fly.io», и
`docs/tasks/AAB-3.md`.

Живой пример: [anex-analyst-bot.fly.dev](https://anex-analyst-bot.fly.dev),
приложение `anex-analyst-bot` (дефис — Fly не принимает подчёркивание в имени
приложения, см. `CLAUDE.md`), регион `ams`.

Понадобится [`flyctl`](https://fly.io/docs/flyctl/install/) и аккаунт на Fly.io
(с привязанной картой — Fly требует её для создания приложений, даже в рамках
бесплатных лимитов).

**Важно:** `fly.toml` лежит в `deploy/fly.toml`, а не в корне репозитория.
Команды, которые сами не находят приложение через `--app`/`-a` (`fly deploy`,
`fly status`, `flyctl volumes create`, `fly secrets set` без `--app`),
выполняются из корня репозитория с явным `--config deploy/fly.toml` — иначе
`flyctl` не найдёт конфиг в текущей директории. Команды ниже уже это
учитывают.

### 1. Приложение и volume — один раз, до первого деплоя

```bash
fly apps create <имя-приложения>
```

Пропиши реальные `app` и `primary_region` в `fly.toml` — регион узнать через
`fly platform regions`.

Volume должен существовать **до** первого `fly deploy`, иначе `activity.db`
создастся на эфемерном диске контейнера и пропадёт при следующем деплое:

```bash
flyctl volumes create data --size 1 --region <тот же регион, что в fly.toml> \
  --config deploy/fly.toml
```

Только один volume на приложение — если по ошибке когда-то создались два
(например, при возврате к идее двух process group), лишний нужно удалить
(`flyctl volumes destroy <id>`), иначе `bot` и `dashboard` разъедутся по
разным дискам.

### 2. Память машины — не меньше 512MB

`bot.py` (aiogram + telethon) и `dashboard.py` (streamlit + pandas + altair +
pyarrow) работают в одном контейнере одновременно. На дефолтных 256MB
(`shared-cpu-1x`) машина уходит в нехватку памяти почти сразу после старта —
`MemFree` проседает до считанных мегабайт, health-check и даже SSH перестают
отвечать (не падение с очевидной ошибкой, а тихое зависание). Проверено
живьём: `bot.py` даёт ~90MB RSS, `streamlit` — ~40MB сразу после старта, до
первого реального открытия дашборда. Ставить сразу 512MB:

```bash
flyctl scale memory 512 --app <имя-приложения>
```

### 3. Secrets

`.env` в проде не используется — только для локальной разработки. На Fly
секреты задаются отдельно:

**Не копируй значения из `.env` вместе с кавычками.** В `.env` строки вида
`BOT_TOKEN="..."` — кавычки там для читаемости, `python-dotenv` их сам убирает
при чтении. `fly secrets set KEY=value`, как и `docker --env-file`, кавычки
НЕ убирает — они попадут в значение буквально и токен станет невалидным
(проверено локально: именно так и сломался тестовый `docker run`, пока не
убрал кавычки вручную). Задавай secrets без кавычек:
`fly secrets set BOT_TOKEN=123456:AAExampleTokenValue` — не
`fly secrets set BOT_TOKEN="123456:AAExampleTokenValue"`.

```bash
fly secrets set \
  --config deploy/fly.toml \
  BOT_TOKEN=... \
  OWNER_USER_ID=... \
  TG_API_ID=... \
  TG_API_HASH=... \
  TG_PHONE=... \
  DB_ENCRYPTION_KEY=...
```

Безопаснее — прогнать нужные переменные из `.env` напрямую в `fly secrets
import` через редирект, не печатая значения в терминал:

```bash
grep -E '^(BOT_TOKEN|OWNER_USER_ID|TG_API_ID|TG_API_HASH|TG_PHONE|DB_ENCRYPTION_KEY)=' .env \
  | sed -e 's/^\([A-Z_]*\)="\(.*\)"$/\1=\2/' \
  | flyctl secrets import --app <имя-приложения> --stage
```

`--stage` только сохраняет secrets, не деплоит — следующий `fly deploy`
применит их к машине.

`DATA_DIR` секретом не является — он уже задан в `fly.toml` (`[env]`) и
указывает на точку монтирования volume (`/data`).

### 4. Перенос существующих данных (если есть локальные `activity.db` / session)

Разовая ручная операция. **Важно:** если `history.py` уже успел хоть раз
подключиться на этой машине (например, кто-то раньше вызвал `/import_history`
без готовой сессии), Telethon сам создаёт пустой неавторизованный файл сессии
по этому пути — `put` в sftp-шелле откажется перезаписать существующий файл
(`file exists on VM`). Сначала удалить старый файл через обычный SSH, потом
заливать настоящий:

```bash
fly ssh console -a <имя-приложения> -C "rm /data/history_session.session"
fly ssh sftp shell -a <имя-приложения>
> put activity.db /data/activity.db
> put history_session.session /data/history_session.session
```

(имя session-файла — значение `TG_SESSION` из `.env`, по умолчанию
`history_session`). Если это первый запуск без существующих данных — шаг
можно пропустить, `activity.db` создастся автоматически.

### 5. Деплой

```bash
fly deploy --config deploy/fly.toml
```

### 6. Проверка

- `fly status --config deploy/fly.toml` — **одна** машина (`app`) в
  `started`, health-check `1/1`.
- `/status` в тестовом чате отвечает; `/import_history` в личке с ботом
  проходит без просьбы о повторной авторизации Telethon-сессии.
- Дашборд открывается по адресу приложения и видит данные, которые собрал
  `bot`/`history.py` (общий volume — общие данные по определению).
- **Fail-fast:** найти PID `bot.py` внутри контейнера и убить его вручную —
  вся машина должна упасть и **сама подняться обратно** (не просто упасть):

  ```bash
  fly ssh console -a <имя-приложения> -C "..." # найти PID python (bot.py) через /proc
  fly ssh console -a <имя-приложения> -C "bash -c 'kill -TERM <pid>'"
  fly checks list -a <имя-приложения>  # через несколько секунд снова passing
  ```

  Если машина падает, но НЕ поднимается сама — проверить `[[restart]]` в
  `fly.toml` (`policy = "always"`). Без этого блока Fly считает чистый
  `exit 0` намеренной остановкой и не перезапускает машину — реально
  воспроизводилось при живом тесте.
- `fly logs -a <имя-приложения>` не показывает пауз/пропусков между
  сообщениями за несколько часов теста (long polling не должен прерываться).

### После первого деплоя

Повторные `fly deploy` не требуют пересоздания volume и secrets — только
пересобирают образ и перезапускают машину. `activity.db` и session-файл
переживают редеплой, потому что живут на volume, а не в образе.

## Данные в базе

**`messages`** — `message_id`, `chat_id`, `user_id`, `username`, `full_name`,
`text_encrypted` (шифротекст), `msg_type`, `timestamp` (UTC), `weekday`
(0=понедельник), `hour`.

**`reactions`** — `chat_id`, `message_id`, `reaction_count`, `updated_at`.

**`chats`** — `chat_id`, `title`, `updated_at`, `active`, `activated_by`,
`activated_at`, `deactivated_at`.

## Важные нюансы

- **Метрики активности (сводка, графики, топ участников) — по умолчанию только
  текстовые сообщения** (переключается в сайдбаре). Реакции и разбивка по типам
  всегда считают по всем типам сразу.
- **Метрика "популярности"** сообщения — количество реакций. Если понадобятся
  реплаи вместо/вместе с реакциями — это отдельный запрос по тому же принципу
  в `db.py`.
- **Реакции обновляются только вручную, через `/import_history`.** Постоянного
  фонового пересчёта (как раньше в `collector.py`) больше нет — `history.py`
  подключается только на время команды. При каждом `/import_history` заново
  пересчитываются реакции у сообщений за последние 3 дня (независимо от того,
  что реально изменилось) — если нужны более свежие реакции по старым
  сообщениям, придётся дождаться следующего ручного запуска.
- **Если у тебя уже была база до этого обновления** — формат хранения текста
  поменялся (`text` → `text_encrypted`, шифрование), это несовместимое изменение
  схемы. Самый простой способ перейти: удалить `activity.db` и запустить сбор
  заново — бот перекачает историю целиком в новом формате. Заодно чинится более
  ранний баг классификации типа сообщения (фото/видео с подписью раньше
  ошибочно считались `text`).
- **Если MTProto блокируется провайдером** — в `history.py` уже используется
  `system_version="4.16.30-vxCUSTOM"`, который решил эту проблему в вашем случае.
  Если понадобится прокси — добавь `proxy=(...)` при создании `TelegramClient(...)`.
- **Параллельный доступ к базе** — `bot.py` включает WAL-режим SQLite при
  старте, дашборд подхватывает её автоматически.

## О приватности собираемых данных

- **Двойное согласие.** Сбор требует и добавления бота в чат, и явного
  `/activate` от админа (или владельца бота) — до этого момента бот ничего не
  пишет, хотя и состоит в чате. `/status` доступен любому участнику, чтобы
  узнать о сборе не спрашивая админа. Если бота удалили из чата или он потерял
  доступ — сбор автоматически помечается остановленным.
- **Что защищено сейчас:** текст сообщений на диске нечитаем без ключа. Это
  главная угроза, о которой стоило беспокоиться — именно текст раскрывает
  содержание переписки.
- **Что не защищено:** метаданные (кто, когда, кому, сколько реакций) хранятся
  открытым текстом — без них не работает вообще ничего в дашборде, а сами по себе
  они не раскрывают содержание. Сам файл `activity.db` тоже не зашифрован целиком:
  если нужна защита и на этом уровне (например, от кражи ноутбука целиком) —
  следующий шаг — переезд на SQLCipher вместо обычного `sqlite3`, отдельная
  утилита с собственным C-расширением, сознательно не стала тащить её сразу
  из-за более хрупкой установки (нужен системный SQLCipher), сначала оценили
  риск как менее приоритетный, чем защита самого текста.
- **Парольный режим — не многопользовательский.** Если дашборд когда-нибудь
  станет общим для нескольких человек одновременно (не то, для чего он сейчас
  сделан), текущая реализация не изолирует пароль по сессиям пользователей —
  учти это, если планируешь такой сценарий.
