# ANEX-005 — Реорганизация репозитория

Передать сеньору в VS Code. Первая задача по новому стандарту именования —
дальше все TASK-файлы называются `ANEX-XXX.md` и лежат в `docs/tasks/`.
Существующие `TASK-phase1-bot.md`, `TASK-phase2-hosting.md`,
`TASK-phase2-correction.md` переименовываются в `ANEX-001.md`/`ANEX-002.md`/
`ANEX-003.md` в рамках этой же задачи (используй `git mv`, если репозиторий
под git — сохранит историю).

Никакой логики не меняется, только расположение файлов и (там, где неизбежно)
пути в конфигах. Если что-то в процессе требует менять больше, чем перечислено
ниже — остановиться и свериться с архитектором, не импровизировать.

## Порядок работ

### Шаг 1. Низкий риск: задачи и деплой-конфиги

- `mkdir -p docs/tasks deploy`
- `git mv TASK-phase1-bot.md docs/tasks/ANEX-001.md`
- `git mv TASK-phase2-hosting.md docs/tasks/ANEX-002.md`
- `git mv TASK-phase2-correction.md docs/tasks/ANEX-003.md`
- `git mv Dockerfile deploy/Dockerfile`
- `git mv fly.toml deploy/fly.toml`
- `git mv start.sh deploy/start.sh`

Ничего из этого не исполняется в рантайме сразу — можно делать, не боясь
уронить прод, но следующий шаг (Docker/Fly пути) обязателен, иначе деплой
сломается при следующем `fly deploy`.

### Шаг 2. Код в `src/`

- `git mv bot.py crypto.py dashboard.py db.py generate_key.py history.py src/`
- **Импорты внутри файлов НЕ трогать.** Модули импортируют друг друга как
  соседи (`import db`, `from crypto import ...` и т.п.) — это продолжит
  работать без изменений, если модули запускаются как `python src/bot.py`
  (Python кладёт директорию самого скрипта, `src/`, в начало `sys.path`
  независимо от текущей рабочей директории запуска). Не переходить на
  `python -m src.bot` и не добавлять `src/__init__.py` — это отдельная,
  более инвазивная схема с пакетными импортами, здесь не нужна.
- Все места, где раньше был вызов `python bot.py` / `streamlit run
  dashboard.py` / `python generate_key.py` (README, `start.sh`) — заменить на
  `python src/bot.py` / `streamlit run src/dashboard.py` / `python
  src/generate_key.py`, запуская **из корня репозитория** (не `cd src`
  внутри), чтобы `.env` в корне продолжал находиться штатной загрузкой
  переменных окружения (`python-dotenv` ищет `.env`, поднимаясь от текущей
  директории — если `cd src`, поиск может не дойти до корня в зависимости от
  того, как именно вызывается `load_dotenv()`).

**Проверка:** локально, из корня репозитория — `python src/bot.py` и
`streamlit run src/dashboard.py` стартуют без `ImportError` и без ошибок
чтения `.env`.

### Шаг 3. Docker и Fly — синхронизация путей

- `deploy/Dockerfile`: пути в `COPY` (`requirements.txt`, `src/`, `deploy/
  start.sh` и т.п.) обновить под новую структуру. Контекст сборки
  (`docker build` / `fly deploy`) остаётся корнем репозитория — Dockerfile
  просто переехал в подпапку, это не меняет, что видно сборке, но
  относительные пути **внутри** Dockerfile теперь считаются от контекста
  (корня), а не от `deploy/`, если явно не указано иное — проверить, что
  `COPY src/ src/` (а не `COPY ../src/ src/`) резолвится верно.
- `fly.toml` переехал в `deploy/fly.toml` — команды `fly deploy`/`fly status`
  и т.п. из корня репозитория теперь требуют `--config deploy/fly.toml`
  (или Fly сам предложит это при отсутствии `fly.toml` в текущей директории —
  проверить фактическое поведение). Задокументировать явно в README, чтобы
  при следующем деплое (не только сейчас) не забыть флаг.
- `fly.toml`: секция `[build]` — путь до `dockerfile` обновить на
  `deploy/Dockerfile`, если путь был указан явно.
- `deploy/start.sh`: команды запуска (`python bot.py`, `streamlit run
  dashboard.py`) — заменить на пути из `src/` (см. Шаг 2). Само расположение
  `start.sh` внутри контейнера (то, что вызывает `Dockerfile` как
  `ENTRYPOINT`/`CMD`) — сверить, что путь до него в `Dockerfile` тоже
  обновлён под `deploy/start.sh` как источник при `COPY`.

**Проверка:** `docker build` из корня репозитория проходит без ошибок путей.
Реальный `fly deploy` (или как минимум `fly deploy --config deploy/fly.toml
--build-only`/dry run, если доступен) — не показывает ошибок конфигурации.

### Шаг 4. Данные — самый чувствительный шаг

`activity.db`, `activity.db-shm`, `activity.db-wal`, `history_session.session`
— это реальные продакшн-данные и авторизованная Telegram-сессия. Прод на Fly
их не касается напрямую (там свой физический volume, смонтированный отдельно
от этого локального репозитория) — риск целиком в локальной среде разработки
и в том, чтобы не разъехались пути между локальным запуском и тем, что уже
задеплоено.

1. **Сначала бэкап.** Скопировать все четыре файла в отдельное место вне
   репозитория, прежде чем что-либо трогать.
2. **Проверить, как сейчас читается путь к БД/сессии** в `db.py`/`crypto.py`/
   `history.py`: если путь уже параметризован переменной окружения (что
   предполагалось ещё в фазе 2 для Fly-деплоя) — использовать её же, просто
   сменив локальное значение по умолчанию на `data/`. Если путь захардкожен
   как голое имя файла (`"activity.db"`) — параметризовать сейчас (единая
   константа/переменная окружения, например `DATA_DIR`, дефолт `data/` для
   локальной разработки).
   **Важно:** значение, которое реально используется на Fly (путь к
   существующему смонтированному volume), **не менять**. Эта задача трогает
   только локальную разработку — прод-конфигурация Fly (secrets/env,
   `[mounts]` в `fly.toml`) остаётся как есть.
3. `mkdir -p data`, `git mv activity.db activity.db-shm activity.db-wal
   history_session.session data/`.
4. Проверить `.gitignore` — `data/` (или конкретно эти файлы внутри неё)
   должен быть игнорируемым, как и раньше были игнорируемы `activity.db*` и
   `*.session` в корне. Секреты и живые данные в git не должны попадать ни в
   старом, ни в новом расположении.

**Проверка:** локально — бот подключается к `data/activity.db`, видит
существующие записи (не создаёт пустую новую базу вместо реальной!); дашборд
открывает те же данные; `/import_history` находит и использует
`data/history_session.session` без повторной авторизации.

**Не деплоить это изменение на Fly, пока не пройдена локальная проверка
целиком.** Если после локальной проверки решено катить на прод — это
отдельный явный шаг с бэкапом продовой БД перед деплоем, не автоматическое
продолжение этой задачи.

## Не делать в этой задаче

Не переходить на пакетные импорты (`src/__init__.py` + `from src.db import
...`) — обычные соседние импорты внутри `src/` достаточны и менее рискованны.
Не удалять и не переименовывать ничего, кроме перечисленного выше. Не менять
прод-конфигурацию Fly (значения secrets, путь монтирования volume) в рамках
этой задачи.
