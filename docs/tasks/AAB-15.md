# AAB-15 — Уведомление новых участников про `/dashboard`

Найдено в ходе AAB-14: новый участник уже активированного чата никак не
узнаёт о существовании команды `/dashboard` — бот реагирует только на смену
своего собственного статуса (`on_my_chat_member`), не на присоединение
обычных участников. Проактивно написать в личку новому человеку бот не
может технически (Telegram не разрешает ботам писать первыми тому, кто не
начинал с ними диалог) — решение ниже это учитывает.

## Решение

Добавить обработчик `new_chat_members` (`bot.py`). При появлении новых
участников в чате, где сбор **сейчас активен** (`is_chat_active(chat_id)`):
опубликовать в самой группе (не в личку — технически невозможно первым, и
это соответствует уже существующему принципу «Прозрачность для участников»)
короткое сообщение вида: «Добро пожаловать, [имена новых участников]! В
этом чате идёт сбор статистики (`/status` — подробности). Личный дашборд —
`/dashboard` в личке с ботом.»

## Детали

- Если сбор в чате **не активен** — ничего не публиковать (см. существующий
  инвариант: сбор и всё, что с ним связано, гейтится через
  `is_chat_active`).
- Если событие `new_chat_members` включает нескольких человек разом
  (добавили пачкой) — одно сообщение на всех, не по одному на человека.
- Если среди «новых участников» оказывается сам бот — это не тот случай,
  это существующий флоу `/activate` через `on_my_chat_member`, исключить из
  этого обработчика, не дублировать.
- Текст сообщения — не жёсткий, смысл сохранить (приветствие + напоминание
  про сбор + как получить личный доступ).

## Критерии готовности

- [x] Новый участник активного чата получает публичное сообщение в группе
      с упоминанием `/dashboard`.
- [x] Несколько участников разом — одно сообщение, не спам по одному.
- [x] В неактивном чате — сообщение не публикуется.
- [x] Добавление самого бота не триггерит это сообщение повторно поверх
      существующего флоу `/activate`.
- [x] Обновлены «Правила и инварианты» в `CLAUDE.md` (новый пункт про
      уведомление новых участников — расширение принципа «Прозрачность для
      участников», не отдельная тема).

## Итог

Добавлен обработчик `on_new_chat_members` (`bot.py`, `F.new_chat_members`),
зарегистрирован до catch-all `collect_message`, чтобы не дублировать запись
служебного сообщения как обычной активности. Логика: исключить из списка
самого бота (`user.id == bot.id`) — если после исключения список пуст,
ничего не публиковать (это чистый `/activate`-флоу через
`on_my_chat_member`); иначе проверить `is_chat_active` и, если чат активен,
опубликовать одно сообщение на всю пачку новых участников, `/status` и
`/dashboard` каждая на отдельной строке (см. правило проекта про переносы
команд в сообщениях бота).

**Доработка по итогам первого живого теста (в рамках той же задачи, без
нового тикета — прямое продолжение уже одобренного принципа «Прозрачность
для участников»):**
- Упоминание участников — через `aiogram.utils.formatting.TextMention`
  (entity `text_mention`, `tg://user?id=...`), а не голым текстом с именем.
  Пингует участника гарантированно, независимо от того, есть ли у него
  публичный `username` — подпись при этом приоритетно `@username`, если он
  есть, иначе `full_name`/`user_id` (`_mention_label`).
- `cmd_activate` (`/activate`) тоже теперь явно упоминает `/dashboard` в
  групповом сообщении при запуске сбора — раньше оно адресовало ссылку
  только тому, кто вызвал команду, и участники, уже бывшие в чате на
  момент активации, не имели способа узнать про личный дашборд вообще (тот
  же пробел, что и у новых участников, просто с другой стороны). Общий
  текст `dashboard_for_others` добавлен в оба варианта ответа (успешная и
  неуспешная отправка ссылки инициатору).

Протестировано вручную дважды: (1) 4 сценария на текстовой версии
сообщения — несколько новых участников, бот+участник вместе, только бот,
неактивный чат; (2) после перехода на `TextMention` — отдельно проверено
построение `text_mention`-сущностей на реальных `aiogram.types.User`
(с `username` и без), офсеты/длины корректны, исключение бота работает
и в этой версии. Оба прогона — на временной SQLite-базе через `db.py`
напрямую, без реального Telegram API. `CLAUDE.md` обновлён — новый абзац
внутри существующего пункта «Прозрачность для участников» в разделе
«Правила и инварианты». Задеплоено на прод (`fly deploy`).
