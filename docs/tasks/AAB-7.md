# AAB-7 — Инициализация git и первый push на GitHub

Выполняется после AAB-6 (унификация имени) — репозиторий на GitHub сразу
создаётся с финальным именем, не переименовывается потом.

Передать сеньору в VS Code. Предпосылка: пользователь уже создал **приватный**
пустой репозиторий `anex_analyst_bot` на GitHub (без README/.gitignore/license)
и авторизован локально (`gh auth login` или SSH-ключ) — это сделано вручную,
до начала этой задачи. Если предпосылка не выполнена — остановиться и
попросить пользователя завершить эти два шага, не пытаться создавать
репозиторий или логиниться от его имени.

## Цель

Безопасно перевести локальный проект под git-контроль и запушить на GitHub —
безопасно означает в первую очередь: секреты и живые данные (`.env`,
`data/activity.db*`, `data/history_session.session`) никогда не должны
оказаться в истории коммитов, даже временно.

## Порядок работ

### Шаг 1. Проверить, не инициализирован ли git уже

- `git status` в корне репозитория.
- Если git уже инициализирован (есть `.git/`) — **не считать текущий
  `.gitignore` достаточной гарантией**. `.gitignore` действует только на
  будущее, он не ретроактивно чистит историю. Проверить:
  - `git log --oneline --all` — есть ли вообще предыдущие коммиты.
  - `git log --all --full-history -- .env data/ '*.session' 'activity.db*'`
    — попадали ли когда-либо эти пути в историю.
  - `git ls-files` — что реально отслеживается прямо сейчас.
- **Если найдены секреты/данные в истории** — не продолжать самостоятельно.
  Сообщить пользователю и архитектору: потребуется `git filter-repo` (или
  BFG Repo-Cleaner) для очистки истории до пуша, это отдельное решение, не
  часть этой задачи по умолчанию.
- **Если истории нет или она чистая** — переходить к шагу 2.

### Шаг 2. Аудит `.gitignore` перед первым коммитом

Свериться, что `.gitignore` явно покрывает (проверить и дополнить, если
чего-то не хватает):

- `.env` (реальные секреты)
- `data/` целиком (или явно `data/activity.db*`, `data/history_session.session`)
- `__pycache__/`, `*.pyc`
- `.venv/` / `venv/` (если виртуальное окружение когда-либо создавалось внутри
  репозитория, а не рядом с ним)
- `.DS_Store`

**Проверка перед коммитом:** `git add -A --dry-run` (или `git status`) и
визуально просмотреть список — ни одного из перечисленных выше файлов не
должно быть в staged/untracked-to-be-added. Если что-то из этого списка
всё же показывается как готовое к коммиту — остановиться и поправить
`.gitignore` до `git add`, не после.

### Шаг 3. Первый коммит и push

- `git init` (если не был инициализирован).
- `git add -A`.
- Ещё раз `git status` — финальная визуальная проверка списка файлов перед
  коммитом.
- `git commit -m "Initial commit"` (или более содержательное сообщение по
  вкусу).
- `git remote add origin <URL приватного репозитория anex_analyst_bot, даст
  пользователь>`.
- `git branch -M main` (если текущая ветка называется иначе).
- `git push -u origin main`.

### Шаг 4. Проверка результата

- Открыть репозиторий на GitHub (в браузере или `gh repo view --web`) и
  визуально убедиться: `.env`, `data/` — отсутствуют; `src/`, `deploy/`,
  `docs/tasks/`, `CLAUDE.md`, `README.md` — присутствуют.
- Убедиться, что репозиторий действительно приватный (`gh repo view` покажет
  `visibility: PRIVATE`, либо проверить на странице настроек репо).

## Критерии готовности

- [ ] История git (новая или существующая) не содержит `.env`/`data/*` ни в
      одном коммите.
- [ ] `.gitignore` покрывает все чувствительные и генерируемые файлы.
- [ ] Репозиторий `anex_analyst_bot` запушен на GitHub, приватный.
- [ ] На GitHub видны `src/`, `deploy/`, `docs/tasks/`, `CLAUDE.md`,
      `README.md`, `requirements.txt`, `.env.example`.
- [ ] На GitHub НЕ видны `.env`, `data/activity.db*`, `data/history_session.session`.

## Не делать в этой задаче

Не создавать репозиторий на GitHub от имени пользователя, не настраивать
GitHub Actions/CI, не трогать сам Fly-деплой. Если в истории git найдены
секреты — не чистить историю самостоятельно (`filter-repo`/BFG) без
явного отдельного решения архитектора и пользователя, просто остановиться
и сообщить.
