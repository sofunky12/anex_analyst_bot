# AAB-11 — Смена формата задач на AAB-N + два новых обязательных правила процесса

Организационная задача, **не про код**. Выполнить **первой**, до AAB-9 и
AAB-10 — они уже написаны в новом формате и ссылаются на правила из этой
задачи.

## Часть 1 — новый формат номеров задач

Было: `docs/tasks/ANEX-XXX.md` с ведущими нулями (`ANEX-001`, `ANEX-008`).
Стало: `docs/tasks/AAB-N.md`, **без ведущих нулей** (`AAB-1`, `AAB-8`,
`AAB-42`, `AAB-999`...). Причина смены: формат с фиксированной шириной
(3 цифры) упирается в потолок на 999-й задаче — без нулей потолка нет.
`AAB` — по первым буквам `Anex Analyst Bot`, не аббревиатура какого-то
нового термина, повода для путаницы с названием проекта нет.

Нумерация сквозная, **как и раньше**: не пропускаем номера, не переиспользуем.
Существующие номера просто переименовываются 1-в-1, счётчик не сбрасывается:

- `ANEX-001.md` → `AAB-1.md`
- `ANEX-002.md` → `AAB-2.md`
- `ANEX-003.md` → `AAB-3.md`
- `ANEX-004.md` → `AAB-4.md`
- `ANEX-005.md` → `AAB-5.md`
- `ANEX-006.md` → `AAB-6.md`
- `ANEX-007.md` → `AAB-7.md`
- `ANEX-008.md` → `AAB-8.md`

(Если в `docs/tasks/` физически лежат не все файлы из списка — переименовать
только то, что реально существует, не создавать недостающие задним числом.)

### Порядок работ

1. `git mv docs/tasks/ANEX-0XX.md docs/tasks/AAB-N.md` для каждого файла из
   списка выше (`git mv`, не удаление+создание — сохраняет историю файла).
2. В каждом переименованном файле поправить заголовок первой строки:
   `# ANEX-0XX — ...` → `# AAB-N — ...` (только номер, остальной текст
   заголовка не трогать).
3. Найти и поправить упоминания старых номеров (`ANEX-001`...`ANEX-008`) в
   `CLAUDE.md`, `docs/PROJECT_CONTEXT.md` (если уже перенесён) и внутри
   других файлов `docs/tasks/*.md`, если они ссылаются друг на друга по
   номеру — заменить на новые (`AAB-N`), само содержание не переписывать.
4. В `CLAUDE.md`, раздел «Структура репозитория», абзац про нумерацию задач
   (сейчас начинается с «Задачи для сеньора — `docs/tasks/ANEX-XXX.md`.
   Сквозная нумерация...») — переписать под новый формат `AAB-N.md` и
   актуальный список соответствий (что есть что), без ведущих нулей.
5. Новые задачи от архитектора (`AAB-9`, `AAB-10`, далее) уже приходят в
   новом формате — просто принять как есть, ничего конвертировать не нужно.

## Часть 2 — подтверждение перед закрытием задачи

**Новое обязательное правило, действует для всех задач начиная с этой.**
Раньше сеньор закрывал задачу по своему усмотрению, как только считал её
выполненной. Теперь — так:

1. Сеньор доводит работу до состояния «критерии готовности выполнены».
2. **Прежде чем помечать задачу закрытой**, сеньор явно спрашивает
   пользователя в чате: «Задачу [AAB-N] закрываем?» — и ждёт ответа.
3. Только после подтверждения — сеньор сам заполняет итоговый контекст
   внутри файла задачи (что сделано, что нашли, отклонения от плана, если
   были — по аналогии с тем, как уже оформлен отчёт в AAB-8) и коммитит
   это вместе с остальными изменениями задачи, затем пушит.
4. Если пользователь просит доработать — не закрывать, вернуться к пункту 1.

Добавить это правило в `CLAUDE.md`, раздел «Рабочий процесс», отдельным
пунктом.

## Часть 3 — обязательная проверка на утечку секретов перед push

**Новое обязательное правило, действует для всех задач начиная с этой, не
только для задач по безопасности.** Перед **любым** `git push` (не только в
рамках задач, явно посвящённых секьюрити):

1. Проверить, что в коммитах, которые вот-вот уйдут в push, нет значений
   секретов — `git diff --staged` (или `git show` на уже сделанные, но ещё
   не запушенные коммиты) визуально проверяется на предмет значений из
   `.env` (`TG_SESSION`, `TG_PHONE`, `TG_API_ID`, `TG_API_HASH`,
   `BOT_TOKEN`, `OWNER_USER_ID`, `DB_ENCRYPTION_KEY`, `DASHBOARD_PASSWORD`,
   `GEMINI_API_KEY` при появлении) — не сами имена переменных (они и так
   есть в коде/`.env.example`), а именно значения.
2. Проверить, что в собственном рабочем выводе инструментов за время задачи
   (терминал, логи отладки) секреты не печатались буквально — если
   печатались, это не блокирует коммит сам по себе (см. принцип «секреты не
   печатаются в вывод» в разделе «Безопасность»), но должно быть явно
   отмечено в отчёте пользователю как потенциальный инцидент, а не
   замалчиваться.
3. Если что-то найдено — **не пушить**, сообщить архитектору через
   пользователя, дальше действовать по уже существующему правилу (не
   чистить историю самостоятельно, ждать решения).
4. Этот шаг — часть обычного рабочего цикла для каждой задачи, не
   отдельный чек-лист, который можно пропустить для «некритичных» правок.

Добавить это правило в `CLAUDE.md`, раздел «Безопасность» → «Принципы»,
рядом с существующим принципом про непечатание секретов в вывод — это его
логическое продолжение (не только не печатать, но и явно перепроверять
перед каждой выкладкой).

## Не делать в этой задаче

Не менять сам принцип сквозной нумерации (без пропусков, без переиспользования)
— меняется только формат записи номера, не правило нумерации. Не трогать
содержание задач при переименовании — только номер в имени файла и в
заголовке.
