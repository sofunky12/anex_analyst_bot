> пишет `bot`. Предложенное решение (не применено, ждёт подтверждения
> архитектора): вместо двух `[processes]`/двух машин — одна машина, оба
> процесса внутри неё (обёрточный скрипт запуска), volume общий по
> определению. Компромисс: `dashboard` теряет автостоп (живёт вместе с
> `bot`, который и так не может засыпать). Остановлено здесь до решения
> архитектора — см. CLAUDE.md, раздел «Хостинг — Fly.io», требует правки.

## Цель

Перевести бота, дашборд и БД с локального запуска на Fly.io: два постоянных
процесса из одного образа, персистентный диск под `activity.db` и Telethon
session, ключ шифрования — в secrets хостинга вместо `.env`.

## Порядок работ

---

### Шаг 1. Dockerfile

- Один образ для обоих процессов (`bot` и `dashboard` запускаются из него по
  разным командам, см. `fly.toml`).
- Базовый образ — Python-версия, реально используемая в проекте (уточнить у
  `python3 --version` в текущем окружении, зафиксировать в Dockerfile и в
  `CLAUDE.md`, если отличается от указанной там 3.9).
- Копирует только нужное для рантайма (`requirements.txt` отдельным слоем для
  кеша сборки).
- НЕ копирует `.env`, `activity.db`, session-файл в образ — они живут на
  volume, не в образе.

**Проверка:** локальный `docker build` проходит, `docker run` с примонтированным
локальным `activity.db` поднимает и бота, и дашборд без ошибок импорта.

---

### Шаг 2. `fly.toml` — два процесса

- `[processes]`: `bot = "python bot.py"`, `dashboard = "streamlit run dashboard.py --server.port 8080 --server.address 0.0.0.0"`.
- Для `bot`: `auto_stop_machines = false`, `auto_start_machines = true` — у
  него нет HTTP-порта (long polling), автостоп по трафику для него не
  применим и опасен (бот будет засыпать и пропускать сообщения).
- Для `dashboard`: обычный HTTP-сервис, автостоп можно оставить включённым.
- Оба процесса — один Fly app, общий volume и secrets (не разносить на два
  отдельных приложения).

**Проверка:** после деплоя `fly status` показывает обе машины (bot, dashboard)
в running; `fly logs -a <app> -i <bot-machine-id>` не показывает засыпаний
между сообщениями в течение тестового периода в несколько часов.

---

### Шаг 3. Persistent volume

- `flyctl volumes create data --size 1 --region <регион>` — **до** первого
  `fly deploy`, не после.
- Точка монтирования — общая для `activity.db` и Telethon session-файла (один
  volume, один mount path на обоих процессах).
- Пути в `db.py`/`crypto.py`/`history.py` к файлу БД и session — через
  переменную окружения (`DATA_DIR` или аналог), не хардкод относительного
  пути, чтобы путь совпадал и локально, и на volume.

**Проверка:** сделать тестовый `fly deploy` дважды подряд — `activity.db` и
session-файл должны сохраниться между деплоями (не пересоздаваться пустыми).

---

### Шаг 4. Secrets и `.env`

- `fly secrets set DB_ENCRYPTION_KEY=... BOT_TOKEN=... OWNER_USER_ID=... TG_API_ID=... TG_API_HASH=...`.
- `.env` полностью убрать из продакшен-пути — остаётся только для локальной
  разработки, в `.gitignore` (уже должен быть).
- Код читает переменные окружения одинаково независимо от источника (Fly
  secrets или локальный `.env` через `python-dotenv`) — не должно быть
  веток кода вида "если на Fly, то так, иначе — так".

**Проверка:** удалить локальный `.env`, поднять контейнер только с
Fly-secrets (или их локальной имитацией через `docker run -e`) — бот и
дашборд стартуют, ключ шифрования подхватывается.

---

### Шаг 5. Первый реальный деплой

- Перенести существующую `activity.db` и Telethon session на volume (одна
  ручная операция — `fly ssh sftp` или аналог, задокументировать команду в
  README, не автоматизировать одноразовую миграцию).
- `fly deploy`.
- Живая проверка: `/status` в тестовом чате отвечает, `/import_history`
  проходит без потери авторизации сессии, дашборд открывается по Fly-адресу.

## Критерии готовности

- [ ] Оба процесса (bot, dashboard) стабильно работают на Fly.io.
- [ ] `activity.db` и session-файл переживают `fly deploy` без потери данных.
- [ ] Бот не «засыпает» между сообщениями (long polling не прерывается автостопом).
- [ ] `DB_ENCRYPTION_KEY` и прочие секреты — только в `fly secrets`, не в
      репозитории и не в образе.
- [ ] README обновлён: команды `flyctl volumes create`, `fly secrets set`,
      перенос существующих данных на volume.
- [ ] Локальная разработка (`.env`) по-прежнему работает без изменений в коде.

необходимость пароля на дашборд раньше срока — остановиться и сверить с
