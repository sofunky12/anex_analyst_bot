# AAB-8 — Аудит после публикации репозитория

**Приоритетная задача** — репозиторий `anex_analyst_bot` только что сделан
публичным (был приватным). Это меняет расчёт риска для всего, что раньше
считалось «низкой экспозицией» — нужна проверка по фактам, не по
предположениям, и желательно как можно скорее.

## Цель

Убедиться, что публикация репозитория не превратила старый (уже закрытый)
инцидент AAB-2/AAB-3 или текущую конфигурацию в реальную, активную утечку.

## Порядок работ

### Шаг 1. Полная проверка истории git (не только текущего состояния)

`.gitignore` защищает только будущие коммиты. Проверить, не попадали ли
когда-либо в **любой** коммит (не только в HEAD):

```
git log --all --full-history --source -- .env data/ '*.session' 'activity.db*'
```

Если список пуст — история чистая, риска нет. Если что-то найдено —
**не пытаться самостоятельно чистить историю** (`git filter-repo`/BFG) без
согласования с архитектором и пользователем, просто зафиксировать: что
именно, в каком коммите, какие пути (не печатать сами значения секретов в
вывод — см. правило в разделе «Безопасность» `CLAUDE.md`, только факт
наличия/пути/коммита).

Отдельно — точечная проверка на конкретные значения, которые упоминались в
инциденте (`TG_API_ID`, `TG_API_HASH`, старые уже отозванные `BOT_TOKEN`/
`DB_ENCRYPTION_KEY`) — если они всё же нашлись в старой истории, это меняет
статус решения «не ротировать `TG_API_ID`/`TG_API_HASH`» с «принято» на
«требует пересмотра» — сообщить архитектору, не решать самостоятельно.

### Шаг 2. Проверка публичной доступности дашборда

- Определить текущий публичный адрес дашборда (`https://anex-analyst-bot.fly.dev`
  или аналог — уточнить через `fly status`/`fly info`).
- Проверить с любого устройства **без** VPN/доступа к внутренней сети Fly,
  открывается ли дашборд по этому адресу без пароля/аутентификации прямо
  сейчас.
- Если открывается без пароля — зафиксировать как активный риск (не
  гипотетический): дашборд показывает реальные метаданные активности чата
  (даже если текст сообщений зашифрован) любому, кто найдёт URL — а теперь
  URL и факт отсутствия пароля вычисляются прямо из публичного кода.

### Шаг 3. Отчёт

Собрать краткий отчёт по обоим пунктам (что нашли / не нашли) — архитектор
на основе отчёта обновит раздел «Безопасность» в `CLAUDE.md` (журнал
инцидентов) и, если понадобится, поставит следующую задачу (ротация,
временное ограничение доступа к дашборду, ускорение фазы 3).

## Критерии готовности

- [ ] Полный вывод `git log --all --full-history` по указанным путям — либо
      пусто, либо задокументированы точные находки (без печати значений).
- [ ] Проверено, доступен ли дашборд публично без пароля прямо сейчас.
- [ ] Отчёт передан архитектору через пользователя, никаких самостоятельных
      действий по ротации/очистке истории не предпринято без согласования.

## Не делать в этой задаче

Не чистить git-историю самостоятельно. Не ротировать `TG_API_ID`/`TG_API_HASH`
самостоятельно — если аудит покажет, что нужно, это отдельное решение
архитектора и пользователя. Не закрывать публичный доступ к дашборду
самостоятельно, если это не тривиально (например, просто временно
остановить машину) — согласовать способ с архитектором, если требуется
что-то серьёзнее.
