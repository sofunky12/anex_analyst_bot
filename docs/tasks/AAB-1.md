# TASK — Фаза 1: бот, выбор чата, догрузка истории

Передать сеньору в VS Code. Архитектурные решения зафиксированы в `CLAUDE.md`
(блок «костыли по процессу с ботом»). Не отступать от них без согласования.

## Цель

Перевести сбор с постоянного Telethon-процесса на постоянного бота (Bot API),
с динамическим выбором чатов через команды и точечной догрузкой истории личным
аккаунтом по требованию.

## Порядок работ

Делать по шагам, не всё сразу. `db.py` первым — от него зависят остальные.

---

### Шаг 1. `db.py` — схема и миграции

- Таблица `chats`: `chat_id` (нормализованный через `get_peer_id`), `title`,
  `active` (bool), `activated_by` (user_id), `activated_at`, `deactivated_at`.
- На `messages` — `UNIQUE(chat_id, message_id)`, запись через `INSERT OR IGNORE`.
- Авто-миграция: `PRAGMA user_version` + список последовательных шагов. Alembic
  не подключать. Существующая `activity.db` должна пережить апгрейд без удаления.
- Хелперы: `activate_chat`, `deactivate_chat`, `is_chat_active`, `list_chats`.
- Существующие строки не переразмечать.

**Проверка:** прогнать миграцию на копии реальной `activity.db` и на пустой;
убедиться, что повторный прогон идемпотентен. Существующие аналитические
запросы после миграции возвращают те же результаты.

### Шаг 2. `history.py` — догрузка истории (заменяет `collector.py`)

- Telethon, `system_version="4.16.30-vxCUSTOM"` при создании `TelegramClient`.
- Session-файл считается уже авторизованным. Если нет — вернуть внятную ошибку
  «нужна ре-авторизация в консоли», не пытаться логиниться интерактивно.
- Async-функция, вызываемая из цикла бота; принимает `chat_id` и callback
  прогресса.
- `get_peer_id()` для нормализации id.
- Классификация `_msg_type`: **медиа проверять раньше text** (иначе фото с
  подписью уедет в "text" — это уже было багом).
- Идемпотентность: повторный запуск не плодит дубли (`INSERT OR IGNORE`).

**Проверка:** импорт в чат, где уже есть live-данные от бота, — количество
строк не должно вырасти на дубликаты.

---

### Шаг 3. `bot.py` — новый, aiogram

Long polling (фаза 1 локальная; webhook — вопрос фазы 2, слой хендлеров держать
транспортно-нейтральным).

Команды:

| Команда | Где | Кто | Действие |
|---|---|---|---|
| `/activate` | группа | админ чата | регистрирует чат, публикует объявление о сборе |
| `/deactivate` | группа | админ чата | останавливает сбор, данные не трогает |
| `/status` | группа | любой участник | показывает, идёт ли сбор |
| `/chats` | лична | владелец | список чатов со статусом |
| `/import_history` | лична | владелец | фоновая догрузка истории |

- Хендлер сообщений: первым делом `is_chat_active(chat_id)`, иначе тихий выход.
- `_msg_type`: медиа раньше text. **Внимание:** у Bot API структура сообщения
  иная, чем у Telethon (`photo`/`video`/`document`/`voice` + `caption`) —
  перенести принцип, а не код один-в-один.
- `my_chat_member`: бота удалили / потерял доступ → авто-деактивация чата.
- `/import_history` — только `OWNER_USER_ID`, запуск как asyncio-задача, прогресс
  в чат, ошибки ловятся и сообщаются (не роняют бота).
- Ключ шифрования: требовать `DB_ENCRYPTION_KEY`, при отсутствии — отказ на
  старте с внятным сообщением.

**Проверка:** реальный бот в тестовой группе. Убедиться, что до `/activate`
сообщения НЕ пишутся в БД, после — пишутся; что `/status` виден обычному
участнику; что `/import_history` от постороннего отклоняется.

## Критерии готовности

- [ ] Миграция проходит на реальной `activity.db` без потери данных.
- [ ] До `/activate` сбор не идёт; после `/deactivate` прекращается; данные целы.
- [ ] Live-сбор и импорт истории в один чат не дают дублей.
- [ ] `/import_history` недоступен никому, кроме `OWNER_USER_ID`.
- [ ] Фото с подписью классифицируется как медиа, не как text — в обоих источниках.
- [ ] Импорт истории не блокирует ответы бота на другие команды.
- [ ] `collector.py` удалён, README описывает privacy mode.

## Не делать в этой задаче

Хостинг, контейнеризация, шаринг дашборда по ссылке, починка сессионности ключа,
смена фреймворка дашборда — это фазы 2-4.
