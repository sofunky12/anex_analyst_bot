# ANEX-006 — Унификация имени: anex_analyst_bot

Заменяет предыдущее содержимое `ANEX-006.md` (git init/push) — та задача
переехала в `ANEX-007.md`, выполняется после этой. Причина порядка: имя
репозитория на GitHub логичнее выбрать правильным один раз, а не
переименовывать после первого push.

## Цель

Устранить расхождение `tg_chat_analytics` / `tg-chat-analytics` — унифицировать
под именем, которое уже носит бот в Telegram: **`anex_analyst_bot`**.

## Важное ограничение платформ — не 100% идентичность, и это нормально

- **Telegram username** (`anex_analyst_bot`) — разрешены только буквы, цифры,
  подчёркивание. Дефис невозможен. Менять не нужно — уже так.
- **Fly.io имя приложения** — наоборот, не принимает подчёркивание (имя
  становится частью домена `*.fly.dev`), только буквы/цифры/дефис. Значит Fly-
  приложение будет называться **`anex-analyst-bot`** (дефис), не
  `anex_analyst_bot`. Это единственное намеренное расхождение, оно объяснимо
  правилом платформы — не путать с прежней случайной нестыковкой
  `tg_chat_analytics`/`tg-chat-analytics`.
- Локальная папка, GitHub-репозиторий, документация — везде `anex_analyst_bot`
  (подчёркивание, как у бота).

## Порядок работ

### Шаг 1. Низкий риск: локальная папка и документация

- Переименовать корневую папку проекта: `tg_chat_analytics` → `anex_analyst_bot`.
  После переименования переоткрыть папку в VS Code (старое окно будет
  указывать на несуществующий путь).
- Пройтись поиском по всему репозиторию (`grep -ri "tg.chat.analytics"` или
  аналог) на упоминания старого имени — в `README.md`, комментариях кода,
  заголовке `CLAUDE.md`, любых бейджах/ссылках — заменить на `anex_analyst_bot`
  (или `anex-analyst-bot`, если контекст — именно Fly, см. выше).
- Проверить, нет ли захардкоженных абсолютных путей со старым именем папки
  где-либо в коде/конфигах (маловероятно в этом проекте, но проверить).

**Проверка:** `grep` по репозиторию не находит остаточных упоминаний старого
имени вне git-истории (которую не трогаем).

### Шаг 2. Высокий риск: перенос Fly-приложения

Текущее приложение `tg-chat-analytics` — живое, с реальными данными
(`activity.db`, авторизованная Telethon-сессия) и настроенными secrets. Fly не
поддерживает переименование приложения на месте — нужно создать новое и
перенести всё руками.

1. **Создать новое приложение:** `fly apps create anex-analyst-bot` (тот же
   регион, `ams`).
2. **Создать volume в новом приложении:** `flyctl volumes create data --size 1
   --region ams -a anex-analyst-bot` (см. костыль про volume в `CLAUDE.md` —
   создаётся до первого деплоя, как и в фазе 2).
3. **Перенести данные со старого volume на новый.** Данные небольшие
   (`activity.db` ~45KB, session ~28KB) — самый простой надёжный путь:
   скачать файлы со старого приложения локально (`fly ssh sftp get` или
   аналог), затем загрузить на новый volume нового приложения (`fly ssh sftp
   shell` после первого деплоя нового приложения, когда volume уже
   примонтирован). Не пытаться переносить volume напрямую между
   приложениями — Fly это не поддерживает.
4. **Скопировать secrets на новое приложение:** `fly secrets set
   DB_ENCRYPTION_KEY=... BOT_TOKEN=... OWNER_USER_ID=... TG_API_ID=...
   TG_API_HASH=... -a anex-analyst-bot` (те же значения, что на старом).
5. **Обновить `deploy/fly.toml`:** `app = "anex-analyst-bot"`.
6. **Остановить бота на старом приложении ПЕРЕД запуском нового.** У бота
   один `BOT_TOKEN` — если старое и новое приложение одновременно держат
   long polling с одним токеном, Telegram вернёт конфликт (`409 Conflict`) и
   оба перестанут нормально получать апдейты. Порядок обязателен: `fly scale
   count 0 -a tg-chat-analytics` (или `fly apps suspend`) → только затем
   деплоить новое приложение.
7. **Деплой нового приложения:** `fly deploy -a anex-analyst-bot` (с уже
   перенесёнными данными на volume).
8. **Живая проверка на новом приложении:** бот отвечает, `/status` работает,
   дашборд показывает перенесённые (не пустые) данные, `/import_history`
   использует перенесённую Telethon-сессию **без повторной авторизации**.
9. **Только после полной проверки — удалить старое приложение:**
   `fly apps destroy tg-chat-analytics`. Не удалять раньше — старое
   приложение с ненарушенным volume это единственный путь отката, если
   перенос данных прошёл не так.

**Проверка:** все критерии готовности фазы 2 (память 512MB, restart policy
`always`, fail-fast скрипт) повторно верны и на новом приложении — они не
переносятся автоматически, `fly.toml` и `Dockerfile` копируются как есть, но
стоит перепроверить `fly status`/логи после первого реального рестарта на
новом имени, а не полагаться, что раз работало на старом — сработает и тут.

## Критерии готовности

- [ ] Папка локально называется `anex_analyst_bot`, документация не содержит
      упоминаний `tg_chat_analytics`/`tg-chat-analytics`.
- [ ] Новое Fly-приложение `anex-analyst-bot` работает, содержит перенесённые
      (не пустые) `activity.db` и Telethon-сессию.
- [ ] `/import_history` на новом приложении не требует повторной авторизации.
- [ ] Старое приложение `tg-chat-analytics` не запущено (остановлено/удалено)
      — не работают оба одновременно с одним `BOT_TOKEN`.
- [ ] `deploy/fly.toml` указывает на `anex-analyst-bot`.
- [ ] Старое приложение удалено только после подтверждённой живой проверки
      нового.

## Не делать в этой задаче

Не менять username бота в BotFather — он уже правильный. Не пытаться
использовать подчёркивание в имени Fly-приложения — платформа его не
принимает, дефис здесь не ошибка, а требование.
