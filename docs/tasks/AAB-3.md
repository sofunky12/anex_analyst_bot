# TASK — Коррекция фазы 2: одна Fly Machine вместо двух processes

Правит `TASK-phase2-hosting.md` (шаги 2 и частично 3). Архитектор согласен с
твоим диагнозом и предложенным фиксом — двигайся по нему. Обоснование и
финальные формулировки — в обновлённом `CLAUDE.md` (костыли про volume,
автостоп, обёрточный скрипт).

## Что меняется относительно исходного TASK

**Было:** `[processes]` с `bot` и `dashboard` как две записи в `fly.toml` →
две машины → два volume.

**Стало:** одна Fly Machine, один process group, оба процесса запускаются
через обёрточный скрипт внутри одного контейнера.

## Шаги

1. **Обёрточный скрипт** (например `start.sh`), запускается как единственная
   команда контейнера:
   - Стартует `python bot.py` и `streamlit run dashboard.py --server.port 8080
     --server.address 0.0.0.0` фоновыми процессами.
   - **Fail-fast:** дожидается завершения любого из двух процессов (`wait -n`
     или эквивалент) и сразу завершается сам — это уронит контейнер и
     заставит Fly перезапустить машину целиком. Не должно быть состояния,
     когда один процесс мёртв, а скрипт продолжает висеть с живым вторым.
   - Прокидывает сигналы (SIGTERM/SIGINT) в оба дочерних процесса при
     остановке машины, чтобы не оставлять зомби-процессы.

2. **`Dockerfile`** — `ENTRYPOINT`/`CMD` меняется на запуск обёрточного
   скрипта вместо прямого запуска одного процесса.

3. **`fly.toml`** — один process group (без деления на `bot`/`dashboard`).
   `auto_stop_machines = false`, `auto_start_machines = true` на весь
   единственный process group (не выборочно, как планировалось раньше) — вся
   машина держится постоянно включённой из-за `bot`.

4. **Volume** — оставить только тот, что уже используется `bot` и содержит
   `activity.db`/session-файл. Второй, авто-созданный под старую схему
   `dashboard`-машины, — удалить (`flyctl volumes list`, затем `flyctl
   volumes destroy <id>` на лишнем). Перед удалением убедиться, что на нём
   нет данных, которые нужно сохранить (по описанию — не должно быть,
   `dashboard` его не успел ничем населить).

5. **Передеплой** и живая проверка: `fly status` показывает одну машину;
   `activity.db` и session видны обоим процессам; убить один процесс вручную
   внутри контейнера — вся машина падает и перезапускается (не остаётся
   наполовину живой).

## Критерии готовности (дополняют исходные)

- [x] `fly.toml` описывает один process group, не два.
- [x] Лишний volume удалён, остался один, с данными на месте.
- [x] `dashboard` реально видит записи, которые пишет `bot` (проверено на
      живых данных — 40 сообщений через `/import_history`, видны в дашборде).
- [x] Падение любого из двух процессов роняет весь контейнер (проверено
      вручную через `kill`), а не остаётся незамеченным.

## Дополнительно найдено и исправлено по ходу (не было в исходном TASK)

- **Память.** Дефолтные 256MB не хватает на `bot`+`dashboard` в одном
  контейнере — машина тихо виснет (health-check и SSH не отвечают, но нет
  явного crash/OOM-kill в логе). Поднято до 512MB (`flyctl scale memory 512`).
  Подробности — в CLAUDE.md, костыль про память/OOM.
- **Restart policy.** Fly не перезапускает машину при чистом `exit 0`
  (дефолт `on-failure`) — а fail-fast через SIGTERM даёт именно `exit 0`.
  Добавлен явный `[[restart]] policy = "always"` в `fly.toml`, подтверждено
  живым тестом (машина падает и сама поднимается за ~1.5с). Подробности —
  в CLAUDE.md, костыль про restart policy.
