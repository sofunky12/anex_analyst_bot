# AAB-10 — Фаза 3: авторизация дашборда с разделением по чатам (приоритетная)

**Приоритетная задача** — по итогам AAB-8 (бывшая ANEX-008) дашборд
публично доступен по `https://anex-analyst-bot.fly.dev` без пароля прямо
сейчас: любой, кто узнает URL, видит метаданные активности **всех** чатов
сразу (кто/когда/тип сообщений/реакции; сам текст остаётся зашифрован).
Архитектор и пользователь решили не тормозить с временными полумерами
(остановка машины, затычка), а сразу закрыть это фазой 3.

**Важное уточнение по ходу обсуждения**: изначально задача предполагала один
общий пароль на весь дашборд. Это не годится — `/activate`/`/deactivate`
доступны админу любого чата, не только владельцу, а значит дашборд может
показывать чаты, которыми управляют разные люди. Один пароль = любой, кто
его получил, видит вообще все чаты, включая чужие. Нужна модель с
разделением: каждый чат/админ видит только свой чат, владелец — видит всё.

## Цель

Два независимых способа входа в дашборд:

1. **Персональная ссылка на конкретный чат** — для админа, который
   активировал сбор в своём чате. Открывает вид только этого чата, без
   доступа к другим.
2. **Мастер-пароль владельца** — открывает вид со всеми чатами и селектором
   между ними (текущий режим, которым пользуется владелец).

Без одного из двух — дашборд не показывает вообще ничего (см. AAB-8).

## Порядок работ

### Часть A — персональные ссылки на чат

1. Добавить в схему БД (`db.py`, авто-миграция, без сноса `activity.db`)
   поле `chat_access_token` в таблице чатов (или отдельную таблицу
   `chat_id → token`) — случайная строка, например `secrets.token_urlsafe(32)`.
2. При `/activate` (`bot.py`) — если токена для этого `chat_id` ещё нет,
   сгенерировать и сохранить. Сформировать ссылку вида
   `https://anex-analyst-bot.fly.dev/?chat=<chat_id>&token=<token>` и
   отправить её **в личные сообщения** тому пользователю, который выполнил
   `/activate` (не в сам групповой чат — иначе увидят все участники).
   Если бот не может написать в личку (пользователь не начинал диалог с
   ботом) — сообщить в группе понятным текстом, что нужно сначала написать
   боту в личку, затем повторить `/activate`.
3. Добавить команду `/dashboard_link` (доступна тому же кругу, что и
   `/activate` — админ чата или владелец) — **перевыпускает** токен для
   текущего чата (старый сразу становится недействительным) и присылает
   новую ссылку в личку. Основной сценарий восстановления при утечке или
   потере ссылки — без этого админ не сможет сам себя защитить.
4. `dashboard.py` — при заходе с параметрами `?chat=<id>&token=<t>` в URL:
   свериться с БД (`hmac.compare_digest` на сам токен, не на весь URL),
   при совпадении показать **только** данные этого `chat_id` — все вкладки
   (Обзор/Участники/Сообщения/Текст, тепловая карта и т.д.) фильтруются по
   нему, без селектора других чатов. При несовпадении/отсутствии —
   ничего не показывать, только экран «нет доступа» (без подсказок, что
   именно не так).

### Часть B — мастер-доступ владельца

5. Отдельный `DASHBOARD_PASSWORD` через `fly secrets set` — **задаёт и
   вводит владелец сам**, значение никогда не сообщается сеньору и не
   попадает в задачу/чат/транскрипт, только имя переменной.
6. Экран входа по паролю (`st.text_input(..., type="password")`,
   `hmac.compare_digest` для сравнения, без логирования значения, без
   query-параметров для самого пароля — только тело запроса Streamlit-сессии
   поверх HTTPS). При успехе — вид со всеми чатами и селектором между ними,
   как сейчас. Успешный вход — в `st.session_state` на время открытой
   вкладки, не в постоянной куке.
7. Если открыт разом и `?chat=...&token=...`, и введён мастер-пароль —
   мастер-режим (всё сразу) имеет приоритет.

### Общее

8. Обновить `CLAUDE.md` (раздел «Безопасность» → «Инциденты», короткая
   запись: дата, что публичный доступ закрыт, добавлена модель с
   разделением по чатам) и «Правила и инварианты» (новый инвариант: доступ
   к дашборду одного чата не даёт доступа к данным других чатов).
9. **Перед push** — проверка на утечку секретов в коммитах (см. AAB-11).
10. Перед закрытием задачи — спросить архитектора/пользователя «задачу
    закрываем?» (см. AAB-11), и только после подтверждения — заполнить
    итоговый контекст в задаче и запушить.

## Критерии готовности

- [x] Без ссылки-с-токеном и без мастер-пароля — дашборд не показывает
      никакого контента, включая метаданные.
- [x] Ссылка на чат А не даёт доступа к данным чата Б (проверено вживую на
      двух разных активированных чатах).
- [x] `/dashboard_link` перевыпускает токен, старая ссылка после этого не
      работает. — **суперсед AAB-12** (см. «Итог»): команда заменена на
      `/dashboard`, поведение (старая ссылка умирает при перевыпуске)
      сохранено и проверено уже в новой модели.
- [x] Мастер-пароль хранится через `fly secrets`, задан пользователем
      напрямую, не через сеньора.
- [x] Сравнение токена/пароля — `hmac.compare_digest`, без логирования
      значений.
- [x] Проверено отсутствие секретов в коммитах перед push.
- [x] Получено подтверждение закрытия задачи перед финальным push.
- [x] Запись в CLAUDE.md о закрытии риска из AAB-8 и новом инварианте
      разделения по чатам.

## Не делать в этой задаче

Не трогать шифрование текста (`DB_ENCRYPTION_KEY`/`DB_ENCRYPTION_SALT`) —
отдельная механика. Не откатывать публичность GitHub-репозитория — решение
уже принято. Не строить полноценную систему аккаунтов/логин через Telegram
Login Widget — обсуждали и осознанно выбрали более простой вариант
(персональная ссылка через личку бота), не переоткрывать этот выбор без
нового обсуждения с архитектором.

## Итог

Выполнено с одним отклонением от плана уже во время реализации плюс
пересмотром модели токена в тот же день отдельной задачей.

**Отклонение от плана.** Изначальная ссылка была `?chat=<chat_id>&token=<t>`
(как явно описано в задаче). По итогам обсуждения рисков URL/логов решили
убрать `chat_id` из адресной строки — сам по себе он не секрет и не даёт
доступа без токена, но незачем светить его лишний раз. Дашборд находит чат
обратным поиском по токену (`hmac.compare_digest` на каждую строку). Старые
ссылки с обоими параметрами продолжают работать — `chat_id` в URL просто
игнорируется, чат определяется только токеном.

**Модель токена пересмотрена в тот же день — AAB-12.** По итогам обсуждения
с пользователем токен на чат целиком (Часть A этой задачи) заменён на токен
пары чат+пользователь, `/dashboard_link` — на `/dashboard` (доступна любому
участнику, не только админу). Решение сформулировано отдельной задачей
через архитектора, не придумано на месте. Часть B (мастер-пароль) не
затронута пересмотром — работает как описано здесь.

Проверено вживую пользователем на двух реальных активированных чатах —
ссылка одного чата не открывала данные другого. Секреты в коммитах —
проверены, не найдены. Подтверждение закрытия получено.
