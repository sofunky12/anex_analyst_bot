# AAB-12 — Доступ к дашборду: команды и токены на пользователя

**Приоритетная задача** — дашборд сейчас публично доступен без пароля (см.
AAB-8, аудит после публикации репозитория) — активный риск.

## Модель токена

Таблица `dashboard_tokens` (`db.py`, авто-миграция): `chat_id`, `user_id`,
`token` (`secrets.token_urlsafe(32)`), `created_at`, уникальный индекс на
`(chat_id, user_id)`. Каждый пользователь получает свою собственную ссылку
на конкретный чат — не общую на весь чат.

## Команды в группе

- **`/activate`** (админ чата или владелец) — включает сбор. Сообщение в
  группе: «Запущен сбор статистики чата. Ссылка на дашборд отправлена вам в
  личку.» Тому, кто вызвал команду, в личку уходит персональная ссылка
  (создаётся токен для пары «этот чат + этот пользователь», если его ещё
  нет). Если бот не может написать в личку — сообщить в группе, что нужно
  сначала написать боту в личку, затем повторить `/activate`.

- **`/deactivate`** (тот же круг) — останавливает сбор. Сообщение в группе:
  «Сбор статистики чата завершён — бот в чате отключён». На токены не
  влияет.

## Команды в личке с ботом

Сообщение со ссылкой (и при `/activate`, и при `/dashboard`) должно
содержать инструкцию: вот ссылка на дашборд; если она потеряется или будет
кем-то расшарена — можно перевыпустить командой `/dashboard`.

- **`/dashboard`** — доступна любому участнику любого чата, без ограничения
  на админа/не админа. Выпускает ссылку для себя, на чат, в котором
  пользователь участвует. Первый вызов создаёт токен и присылает ссылку.
  Повторный вызов перевыпускает: старый токен этой пары немедленно
  перестаёт работать, новый создаётся и присылается. Если пользователь
  состоит в нескольких чатах, где идёт сбор, — сначала предложить выбор
  чата. Список «в каких чатах состоит» — по тем же данным, что и для
  `/access_token` (уникальные `user_id` из уже собранных сообщений по
  каждому `chat_id`).

## Команды только для владельца

- **`/revoke_token [chat_id] [user_id]`** — `user_id` необязателен.
  - Только `chat_id` — отзывает все токены этого чата.
  - `chat_id` + `user_id` — отзывает токен только этого пользователя.
  - Новых токенов не выпускает. Каждому затронутому пользователю в личку:
    «владелец отозвал ваш доступ к дашборду чата [id/название]».

- **`/access_token [chat_id] [user_id]`** — `user_id` необязателен. Доступ
  можно выдать любому участнику чата, не только админам.
  - Только `chat_id` — выпускает и рассылает токены **всем участникам**
    этого чата.
  - `chat_id` + `user_id` — выпускает и присылает ссылку только этому
    пользователю, вне зависимости от статуса.
  - Bot API не даёт получить список всех участников чата напрямую для
    больших/анонимных групп. Практически «все участники» — все уникальные
    `user_id`, встречающиеся в уже собранных сообщениях этого `chat_id`.
    Если личка недоступна для кого-то — пропустить его, не прерывать
    выполнение команды, в конце отчитаться, кому ссылка дошла, а кому нет.

- **`/import_history [chat_id]`** — без изменений.

## Критерии готовности

- [x] Токен привязан к паре `(chat_id, user_id)`, не к чату целиком.
- [x] Тексты сообщений `/activate`/`/deactivate` в группе — по смыслу как
      выше.
- [x] Сообщение со ссылкой (при `/activate` и при `/dashboard`) содержит
      инструкцию про перевыпуск через `/dashboard`.
- [x] `/dashboard` — доступна любому участнику, не только админам; первый
      вызов создаёт, повторный — перевыпускает, рвёт старую ссылку только
      для вызвавшего пользователя.
- [x] `/revoke_token` — оба варианта (с `user_id` и без), уведомление
      затронутым в личку, без автоматического перевыпуска.
- [x] `/access_token` — оба варианта (с `user_id` и без), без ограничения
      по админству.
- [x] `/import_history` — без изменений.
- [x] Мастер-пароль владельца (`DASHBOARD_PASSWORD`) не затронут —
      отдельный, независимый способ входа.
- [x] Обновлены «Правила и инварианты» и «Архитектура» в `CLAUDE.md`.

## Процесс

Перед закрытием задачи — спросить пользователя, закрываем ли. Перед
`git push` — проверить, что в коммитах нет секретов.

## Не делать в этой задаче

Не менять круг лиц с правами на `/activate`/`/deactivate` (админ чата или
владелец). Не трогать мастер-пароль владельца и схему его хранения.

## Итог

Выполнено, плюс два дополнения сверх исходного текста задачи — оба найдены
и запрошены пользователем уже в ходе живого тестирования, не придуманы
сеньором самостоятельно:

1. **Повторный `/activate` на уже активном чате** отвечает «уже запущен...
   `/deactivate`» вместо молчаливой пересылки того же сообщения «Запущен
   сбор...», как будто сбор только что включили — без этого было легко
   решить, что каждый повторный вызов что-то заново инициализирует.
2. **`/deactivate` отзывает все токены дашборда чата** и уведомляет каждого
   затронутого в личку — прямое отклонение от исходного текста задачи
   («на токены не влияет»), добавлено ради быстрого контроля в критичной
   ситуации (например, подозрение на утечку ссылки) одной командой, без
   `/revoke_token` по каждому участнику отдельно. Данные `messages`/
   `reactions` не затрагивает ни в одном из случаев.

Отдельно, в ходе того же тестирования устранён недочёт форматирования:
все сообщения, где упоминается команда для выполнения, теперь кладут её на
отдельную строку — иначе в части клиентов Telegram перенос строки обрезает
или запутывает команду внутри предложения.

Живой тест пользователем: `/activate`, `/dashboard` от не-админа,
`/deactivate` — обнаружено, что фикс с отзывом токенов не сработал на
проде, но причина оказалась не в коде: изменения ни разу не были
задеплоены с самого AAB-10 (последний коммит в git — AAB-9, `fly deploy`
собирает из рабочей директории, а не из git). После `fly deploy` поведение
подтверждено. Секреты в коммитах — проверены, не найдены. Подтверждение
закрытия получено.
