# AAB-16 — Команды по роли, минимизация сообщений в группе, служебные сообщения

Три связанных правки в `bot.py` (команды/сообщения/сборщик), объединены в
одну задачу.

## Часть A — список команд по роли (в личке с ботом)

Новая команда (например, `/help`) в личке — показывает список доступных
команд, разный по роли вызвавшего:

- **Владелец (`OWNER_USER_ID`)** — полный список: `/activate`,
  `/deactivate`, `/status` (в группе), `/dashboard`, `/chats`,
  `/import_history`, `/access_token`, `/revoke_token`.
- **Админ хотя бы одного чата, где бот активен, но не владелец** —
  `/activate`, `/deactivate`, `/status` (в группе), `/dashboard`.
- **Все остальные** — `/dashboard`, `/status` (в группе, если состоит хоть
  в одном).

Определение роли: `user_id == OWNER_USER_ID` → владелец. Иначе — пройтись
по чатам, известным боту, проверить через `bot.get_chat_member(chat_id,
user_id)`, является ли вызвавший админом/создателем хотя бы в одном —
первое совпадение достаточно. Ни то ни другое → обычный пользователь
(безопасный дефолт и для незнакомца).

## Часть B — минимизировать необязательные сообщения в группе

Обязательные уведомления о прозрачности сбора **не трогать**: сообщение при
`/activate` («Запущен сбор...»), при `/deactivate` («Сбор завершён...»),
приветствие новым участникам (AAB-15). Это прямое следствие принципа
«Прозрачность для участников», не переоткрывать.

Всё остальное, что бот пишет в группу сверх этого — сократить до минимума
или перенести в личку, где технически возможно. Известный пример: если бот
не может написать в личку тому, кто вызвал `/activate`, сейчас в группу
уходит развёрнутая инструкция — сократить до нейтрального минимума (короче
текст, не убирать совсем — единственный доступный канал в этом случае).

Прислать архитектору через пользователя короткий список: что нашли, было/
стало по каждому сообщению.

## Часть C — служебные сообщения Telegram и сам бот не должны считаться активностью

**Сначала разобраться, потом чинить** — два симптома ниже могут быть
связаны одной причиной.

**Симптом 1.** После AAB-15 явно исключён только `new_chat_members`. Другие
служебные события (выход участника, закреп, смена названия/фото чата,
миграция и т.п.) могут по-прежнему попадать в `collect_message` как обычные
сообщения. Вместо точечных исключений по одному типу — построить allowlist
реального контента участника (текст, фото, видео, документ, голосовое,
стикер и т.п.), пропускать в сборщик только его.

**Симптом 2.** Пользователь получил сообщение вида «Не удалось отправить
(нет диалога в личке с ботом): 8655523357» — похоже на `user_id` самого
бота. Значит в каком-то перечислении «участников чата» (использовалось в
`/access_token` без `user_id`) попадает сам бот. Проверить, не пишутся ли
собственные исходящие сообщения бота в `messages` (тогда чинится тем же
allowlist'ом из симптома 1); если нет — найти конкретный запрос,
формирующий список участников, и исключить `bot.id` явно.

Если найдены уже существующие записи служебных сообщений/сообщений бота в
БД — сообщить архитектору через пользователя перед любой чисткой данных, не
удалять самостоятельно.

## Критерии готовности

- [ ] Часть A: новая команда выводит список по роли, роль определяется по
      правилам выше.
- [ ] Часть B: обязательные уведомления не изменены; остальные сообщения в
      группе сокращены/перенесены в личку; список изменений передан
      архитектору.
- [ ] Часть C: определён allowlist типов контента; служебные сообщения (все
      типы) не попадают в `messages`; найден и устранён источник появления
      `bot.id` в списке участников для `/access_token`; проверено вживую,
      что бот больше не фигурирует как недоступный получатель.
- [ ] Обновлены «Архитектура» и «Правила и инварианты» в `CLAUDE.md`.

## Итог

**Закрыто пользователем сразу после реализации, до его собственного живого
теста на реальных людях (запланирован на вечер того же дня)** — та же
явная, документированная девиация от обычного порядка «сначала живой тест,
потом закрытие», что и в AAB-15. Если вечерний тест найдёт проблему — фикс
идёт новой задачей, не переоткрытием AAB-16.

**Часть A.** `/help` (личка) — список команд по роли. Роль: `OWNER_USER_ID`
→ владелец; иначе перебор чатов с активным сбором (`db.active_chat_ids`)
через `bot.get_chat_member` → админ/создатель хотя бы в одном → админ;
иначе — обычный пользователь.

**Часть B.** Единственное найденное избыточное сообщение в группе —
разворачивающаяся инструкция в `/activate`, когда боту не удаётся написать
в личку вызвавшему. Сокращено до нейтрального минимума (см. `bot.py`,
комментарий у этой ветки). Остальные групповые сообщения (`/activate`
успех, `/deactivate`, `/status`, приветствие новым участникам AAB-15) —
уже минимальны, это и есть обязательные уведомления о прозрачности, не
трогались.

**Часть C.** Оба симптома подтверждены на реальных локальных данных
(`data/activity.db`, чат «Тест группа»), не только предположительно:
- служебные Telegram-события (не `new_chat_members`, тот уже отфильтрован
  в AAB-15) попадали в `messages` как активность реального участника
  (`msg_type="other"`);
- **собственные исходящие сообщения бота в группу реально доставляются ему
  обратно как `message`-апдейты** — гипотеза «бот не видит свои же
  сообщения» не подтвердилась на данных, это и объясняло появление
  `user_id` бота в списке участников для `/access_token`.

Фикс — allowlist реального content_type участника (`_ALLOWED_CONTENT_TYPES`
в `bot.py`) + явное исключение `user.id == bot_id` в `_save_message`, плюс
защитный фильтр `uid != bot.id` в `/access_token` на случай уже
накопленных старых строк. Уже существовавшие 8 «плохих» строк (4
служебных + 4 от бота) в локальной БД **не удалялись** — решение об
очистке остаётся за архитектором/пользователем, не тронуто самостоятельно.

Обновлён `CLAUDE.md` (архитектура `bot.py`, новый инвариант про allowlist,
запись в «Баги и фиксы» с находкой по бэку собственных сообщений бота).

**Проверено:** `python3 -m py_compile` всех модулей `src/`; точечно —
`db.active_chat_ids`, `Bot.id`, allowlist-логика на реальных локальных
данных. **Не проверено мной** (нет доступа к живому Telegram) — реальный
диалог с ботом; это и есть вечерний тест пользователя.
