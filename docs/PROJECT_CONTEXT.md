# anex_analyst_bot — контекст проекта

Этот документ — для стратегических обсуждений с архитектором (раньше жил
только в project knowledge Claude.ai Projects как отдельный вручную
загруженный файл, теперь переносится в репозиторий задачей AAB-9, чтобы
идти тем же путём, что и `CLAUDE.md`). Здесь фиксируются архитектурные
решения и ход рассуждений, а не код. Конкретная реализация и рабочие
костыли — в `CLAUDE.md` в корне репозитория, это документ для сеньора
(Claude Code), а не для архитектора.

> проект на стадии «Фаза 1 в процессе» — устарела полностью, не упоминала

## Что это за проект

Инструмент для сбора и анализа активности Telegram-чата: кто, когда и что
писал, реакции, распределение по дням/часам/типам сообщений, простой
текстовый анализ, а в перспективе — AI-надстройка поверх метрик. С
шифрованием текста сообщений — принципиальное требование с самого начала.
Проект не ограничен одним чатом владельца — активировать сбор в своём чате
может любой чат-админ, не только владелец (см. ниже про доступ к дашборду).

Сервис называется `anex_analyst_bot` — так же, как бот в Telegram
(`@anex_analyst_bot`). Единственное расхождение — Fly-приложение
`anex-analyst-bot` (дефис вместо подчёркивания, вынужденно платформой Fly).

## Текущее состояние (актуально на 2026-07-21)

**Фаза 1 — аккаунты и выбор чата: завершена, проверена вживую.**
`bot.py` (aiogram, Bot API) — постоянный сбор, команды `/activate`,
`/deactivate`, `/chats`, `/import_history`. `history.py` (Telethon) —
точечная догрузка истории по требованию. Динамический выбор чатов через
команды.

**Фаза 2 — хостинг: завершена, проверена вживую.** Fly.io, одна машина на
`bot.py`+`dashboard.py` (изначальный план с двумя `[processes]` не сработал —
Fly создаёт под каждый process group отдельный volume, шаринг между машинами
невозможен). `shared-cpu-1x:512MB`, `restart.policy = "always"`,
fail-fast обёрточный скрипт.

**Реорганизация и публикация — завершены.** Репозиторий реорганизован
(`src/`, `deploy/`, `docs/tasks/`), унифицировано имя проекта
(`tg_chat_analytics` → `anex_analyst_bot`, включая перенос живого
Fly-приложения), выложен на GitHub — сначала приватно, затем сделан
**публичным** осознанным решением пользователя.

**Аудит после публикации (AAB-8, бывшая ANEX-008) — завершён.** История git
чистая: секреты никогда не коммитились, проверено полным логом
(`git log --all --full-history`) по секретным путям и точечно по именам
переменных. Дашборд при этом оказался **публично доступен без пароля** —
активный риск, не гипотетический: любой, кто узнает URL, видит метаданные
активности чата (не текст — он зашифрован).

**Фаза 3 — авторизация дашборда (AAB-10, бывшая ANEX-010) — в работе,
приоритетная.** Решение: не тормозить временными полумерами (остановка
машины и т.п.), закрыть риск сразу полноценной фазой 3. Модель доступа —
**разделение по чатам**, не общий пароль:
- Персональная ссылка с токеном на конкретный `chat_id`, которую бот
  присылает в личку админу при `/activate`; команда `/dashboard_link`
  перевыпускает токен при утечке/потере.
- Отдельный мастер-пароль владельца — вид со всеми чатами и селектором.

Причина именно такой модели: `/activate`/`/deactivate` доступны любому
чат-админу, не только владельцу — общий пароль означал бы, что любой его
обладатель видит вообще все чаты, включая чужие. Это обсуждалось отдельно и
признано неприемлемым.

**AI Activity Feed (Gemini) — концепт принят, Шаг 1 готов к реализации, не
начат.** Расширение от внешнего RFC (агент Gemini как источник
архитектурной идеи, реализацию делает сеньор). Тон — нейтральный и
профессиональный, не игровой (осознанный отход от исходного RFC, который
предлагал развлекательный стиль). Два контура: метаданный (Шаг 1, работает
всегда, без доступа к `crypto.py`) и текстовый (Шаг 2, требует
аутентифицированной сессии — технически станет возможным после AAB-10, но
включение Шага 2 остаётся отдельным решением, не автоматическим следствием
фазы 3). Обязательные ограничения: обезличивание в ленте и метриках,
анонимизация текста перед отправкой в Gemini, graceful degradation при
сбое API.

## Пройденный путь и ключевые решения

### Сбор данных: бот vs личный аккаунт (MTProto)

Решение: **Bot API — основной, постоянный механизм сбора**, личный аккаунт —
только для точечной догрузки истории по `/import_history`, не висит
постоянно. Это и есть паттерн, под который Telegram проектировал
автоматизацию — прозрачно, безопасно, без вопросов. Bot API принципиально не
читает историю до момента добавления бота — платформенная граница.

### Выбор чата и доступ

Динамически через команды бота: `/activate` (админ чата ИЛИ владелец —
владелец может управлять сбором даже не будучи админом чужого чата),
`/deactivate` (останавливает сбор, данные не трогает), `/chats` и
`/import_history` — строго только владелец (используют его личный
Telethon-аккаунт). Именно из-за того, что `/activate` доступен чужим
чат-админам, доступ к дашборду в фазе 3 разделён по чатам, а не общий.

### Шифрование

`text` → `text_encrypted` через Fernet. Метаданные (кто/когда/тип/реакции)
открыты — на них строится вся аналитика. SQLCipher рассматривали и отклонили
(системная C-библиотека, хрупкая установка). Два режима ключа:
`DB_ENCRYPTION_KEY` (статический) или `DB_ENCRYPTION_SALT` (пароль,
недоступен боту — только дашборду и консоли). Это не имеет отношения к
токену/паролю входа в дашборд из фазы 3 — разные механизмы: один защищает
сам текст сообщений, другой — вход в приложение.

### Хостинг — решено

Fly.io, не VPS: два долгоживущих процесса и два файла, которым нужен
персистентный диск (`activity.db`, Telethon session) — отсекает PaaS с
эфемерной файловой системой; Fly даёт volumes и secrets без необходимости
администрировать ОС, оправдано профилем пользователя (аналитик, не
системный администратор). Перенос приложения вскрыл, что путь `dockerfile`
в `fly.toml` резолвится относительно самого файла, а не корня, и что память
машины не переносится на новое приложение автоматически — детали в
`CLAUDE.md`.

### Дашборд — открытый вопрос по фреймворку

Всё ещё Streamlit, работает, протестирован. Решение по смене фреймворка
(Reflex / FastAPI + фронтенд) по-прежнему отложено до понимания реального
масштаба использования — не пересматривается, пока фаза 3 не пройдена.

## Рабочий процесс (мета-договорённость, обновлено)

- **Архитектор (Claude.ai Projects/Cowork, этот документ + `CLAUDE.md`)** —
  архитектура, смена парадигмы, выбор между вариантами, ревью безопасности.
  Код здесь не пишем и не тестируем.
- **Сеньор (Claude Code в VS Code)** — вся реализация: правки, тесты, запуск.
  Пишет разделы `CLAUDE.md` напрямую по сложившемуся формату — архитектор
  ревьюит через синк/прямое чтение репозитория, не пересобирает файл с нуля
  при каждом изменении.
- **Синхронизация репозитория** — через GitHub-интеграцию Project и, при
  необходимости (когда Project-синк не отдаёт архитектору свежий файл),
  прямое чтение репозитория архитектором по `raw.githubusercontent.com`
  (репозиторий публичный) — рабочий резервный путь, не ручная пересылка
  файлов пользователем.
- Задачи для сеньора — `docs/tasks/AAB-N.md` (формат сменён с `ANEX-XXX` в
  задаче AAB-11 — без ведущих нулей, чтобы не упереться в потолок при
  больших номерах). Сквозная нумерация, без пропусков, без переиспользования.
- **Перед закрытием задачи** сеньор спрашивает пользователя «задачу
  закрываем?» и ждёт подтверждения — только после этого заполняет итоговый
  контекст в файле задачи и пушит (правило введено в AAB-11).
- **Перед каждым `git push`** — обязательная проверка на утечку секретов в
  коммитах и рабочем выводе, не только для security-специфичных задач
  (правило введено в AAB-11).
- На будущее (не сейчас): проект может со временем привлекать несколько
  ИИ-агентов параллельно — например, отдельного агента для data-science/BI
  поверх той же кодовой базы. Понадобится явно развести границы по модулям
  и формат задач между агентами, когда это станет актуальным.

  вперёд без явного согласия, обсуждать концепт до кода.
- Явно заботится о приватности участников чата и безопасности
  автоматизации для Telegram — сквозной приоритет во всех решениях, включая
  новые (обезличивание в AI-ленте, анонимизация перед отправкой в Gemini,
  разделение доступа к дашборду по чатам).
