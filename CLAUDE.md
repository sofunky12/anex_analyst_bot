# anex_analyst_bot — контекст проекта

Сбор активности Telegram-чата + Streamlit-дашборд с шифрованием текста.
Инструкция по установке и запуску — в README.md. Стратегические обсуждения
архитектуры — в Claude.ai Projects (`docs/PROJECT_CONTEXT.md`, синхронизируется
через GitHub), с архитектором. Здесь — решения, ограничения, найденные баги,
правила и вопросы безопасности, которые в код и README буквально не попадают,
но важны сеньору при правках.

> Статус: **Фаза 1** и **Фаза 2** реализованы и проверены вживую. Репозиторий
> реорганизован (ANEX-005), переименован в `anex_analyst_bot` с переносом
> живого Fly-приложения на `anex-analyst-bot` (ANEX-006), выложен на GitHub
> приватно и подключён к Project как источник синхронизации (ANEX-007).
> **AI Activity Feed** — концепт принят, Шаг 1 (метаданные) готов к
> реализации, не начат. Тон фичи — нейтральный/профессиональный, не игровой
> (решение зафиксировано ниже, в разделе AI Activity Feed).

## Структура репозитория

```
anex_analyst_bot/
├── CLAUDE.md              # только в корне — Claude Code читает автоматически отсюда
├── README.md, .env, .env.example, .gitignore, .dockerignore, requirements.txt
├── data/                  # activity.db + Telethon session, gitignored, НЕ в src/

│   └── tasks/             # ANEX-XXX.md — задачи для сеньора
└── src/                   # bot.py, crypto.py, dashboard.py, db.py, generate_key.py, history.py
```

- **Задачи для сеньора — `docs/tasks/ANEX-XXX.md`.** Сквозная нумерация, без
  пропусков и без переиспользования номеров: ANEX-001 = фаза 1 (бот),
  ANEX-002 = фаза 2 (Fly.io), ANEX-003 = фаза 2-коррекция (одна машина),
  ANEX-004 = AI Activity Feed Шаг 1, ANEX-005 = реорганизация папок,
  ANEX-006 = унификация имени, ANEX-007 = git init + push на GitHub,
  ANEX-008 = перенос `PROJECT_CONTEXT.md` в репозиторий.
- **Модули запускаются как `python src/bot.py` и т.п. из корня репозитория**,
  не из `cd src`. Соседние импорты (`import db`, `from crypto import ...`)
  работают без изменений — Python кладёт директорию самого скрипта в
  `sys.path`, переход на пакетные импорты (`src/__init__.py`) не нужен.
- **Путь к `activity.db`/session — параметризован (`DATA_DIR` или аналог),
  не хардкод.** Локальный дефолт — `data/`. Значение на Fly (смонтированный
  volume) реорганизацией репозитория не меняется — это два независимых слоя.

## Архитектура (кто за что отвечает)

- `db.py` — схема SQLite + вся SQL-аналитика, единая точка правды, от неё
  зависят все модули. `msg_type` по умолчанию `"text"` везде, кроме реакций
  и разбивки по типам. Авто-миграция схемы (не нужно удалять `activity.db`
  при изменении схемы). AI Activity Feed добавляет лёгкие агрегирующие
  запросы для аномалий флуда, цепочек реплаев, реактивности и матрицы день×час.
- `crypto.py` — шифрование `text_encrypted` (Fernet), два режима ключа.
  Модуль, который пишет данные, обязан иметь ключ — без него запись
  физически отклоняется.
- `bot.py` — aiogram, Bot API. Постоянно живёт в чате, основной механизм
  сбора. Команды `/activate`, `/deactivate`, `/chats`, `/import_history`.
  Выбор чата динамический через команды.
- `history.py` — точечная догрузка истории личным аккаунтом (Telethon),
  подключается только на время команды `/import_history`, не висит
  постоянным процессом. Заодно пересчитывает реакции за последние 3 дня.
- `dashboard.py` — Streamlit. Только читает БД, никогда не пишет (кроме
  вызовов Gemini API из `ai_analyst.py`). Вкладки Обзор/Участники/Сообщения/
  Текст, тепловая карта день×час, тренд по дням. AI Activity Feed — новая
  вкладка «AI Инсайды».
- `ai_analyst.py` — новый (план), интеграция с `google-genai`. Изолированный
  модуль. Принимает структурированные метаданные из `db.py`, отдаёт
  типизированный JSON для карточек ленты. Не имеет доступа к `crypto.py` на
  Шаге 1.
- `generate_key.py` — утилита генерации ключа шифрования.
- Хостинг — Fly.io, одна машина на оба процесса (`bot.py`+`history.py` по
  требованию и `dashboard.py`), обёрточный скрипт `start.sh`. Машина —
  `shared-cpu-1x:512MB`. Приложение — `anex-analyst-bot`.

## Решения

Осознанные архитектурные выборы и почему не выбраны альтернативы.

- **Bot API — основной сбор, личный аккаунт — только для истории.**
  Постоянный сбор идёт через бота — паттерн, под который Telegram
  проектировал автоматизацию, без вопросов к безопасности. Личный аккаунт
  (Telethon) подключается разово по `/import_history`. Bot API принципиально
  не читает историю до момента добавления бота — платформенная граница, для
  истории нужен `history.py`.

- **Шифруется только поле text, не вся БД.** SQLCipher рассматривали и
  отклонили: системная C-библиотека, хрупкая установка на скудном окружении.
  Метаданные (кто/когда/тип/реакции) открыты — на них строится вся аналитика.

- **Два режима ключа шифрования**: `DB_ENCRYPTION_KEY` (статический, в `.env`
  локально / `fly secrets` на проде) или `DB_ENCRYPTION_SALT` (пароль,
  только для дашборда и консольных запусков — боту недоступен, он
  долгоживущий процесс без интерактивного stdin).

- **Хостинг — Fly.io, не VPS.** Два долгоживущих процесса и два файла с
  нужным персистентным диском (`activity.db`, Telethon session) — отсекает
  PaaS с эфемерной файловой системой. Fly даёт persistent volumes и secrets
  без администрирования ОС — оправдано профилем пользователя (аналитик, не
  системный администратор).

- **Одна Fly Machine на оба процесса, не два `[processes]`.** См. «Платформенные
  ограничения» — исходный план не сработал технически, не по выбору.

- **`/deactivate` только останавливает сбор, данные не трогает.** Удаление —
  отдельной явной командой при необходимости. Случайный `/deactivate` не
  должен стирать историю необратимо.

- **`/import_history` и `/chats` — только владелец деплоя** (`OWNER_USER_ID`).
  Задействует личный Telethon-аккаунт владельца — админ стороннего чата не
  должен иметь возможность это инициировать.

- **`/activate`/`/deactivate` — админ чата ИЛИ владелец, не только админ.**
  Владелец управляет сбором в любом чате, даже не будучи там админом.
  Отдельно от правила выше: `/import_history`/`/chats` — строго только
  владелец, без варианта «или админ».

- **Унификация имени: `anex_analyst_bot` vs `anex-analyst-bot`.** Бот в
  Telegram уже назывался `anex_analyst_bot` (Telegram не разрешает дефис в
  username). Fly.io не разрешает подчёркивание в имени приложения — значит
  там `anex-analyst-bot`. Единственное осознанное расхождение, не путать со
  старой случайной нестыковкой `tg_chat_analytics`/`tg-chat-analytics`.

## Платформенные ограничения

Факты о Telegram/Fly/Streamlit, которые мы не выбирали, а обязаны учитывать.

- **Privacy Mode в BotFather обязан быть выключен.** По умолчанию Bot API
  отдаёт боту только команды и ответы/упоминания — обычные сообщения чата не
  приходят вообще. Нужно `/setprivacy → Disable` **и передобавить бота в
  чат** (смена применяется только при повторном входе). Симптом, если
  забыли: бот «работает», ошибок нет, но в БД почти пусто.

- **MTProto блокируется у пользователя на уровне провайдера.** VPN и
  публичные MTProxy не сработали надёжно. Рабочее решение —
  `system_version="4.16.30-vxCUSTOM"` при создании `TelegramClient` в
  `history.py`. При проблемах с подключением личного аккаунта — начинать с
  проверки этого параметра, а не городить прокси заново.

- **`chat_id` у Telethon и Bot API разный** для супергрупп/каналов (голый id
  vs `-100`-префикс). В одну БД пишут два источника — `bot.py` (Bot API) и
  `history.py` (Telethon), везде обязателен `telethon.utils.get_peer_id()`.

- **Volume на Fly.io физически привязан к одной машине — не шарится между
  несколькими.** Исходный план «`bot` и `dashboard` как два отдельных
  `[processes]`» не работает: Fly создаёт под каждый process group отдельную
  машину и отдельный volume. Исправление: оба процесса — внутри одной Fly
  Machine через обёрточный скрипт.

- **Вся машина не может автостопиться, пока жив `bot`.** `bot` использует
  long polling и не должен засыпать никогда — значит вся машина держится
  постоянно включённой (`auto_stop_machines = false`).

- **Дефолтные 256MB памяти на Fly Machine не хватает на bot+dashboard
  вместе.** ~90MB RSS у бота и ~40MB у дашборда сразу после старта — `MemFree`
  падает до ~2.7MB, машина тихо виснет (не отвечает ни health-check, ни SSH),
  без понятной ошибки. Фикс — `flyctl scale memory 512`. При добавлении
  ещё одного тяжёлого компонента в контейнер (например, `ai_analyst.py`) —
  сразу закладывать память с запасом.

- **Fly не перезапускает машину при чистом `exit 0` без явной restart-политики.**
  Дефолт — `restart.policy = "on-failure"`: код выхода 0 трактуется как
  «намеренная остановка». Fail-fast скрипт при штатном SIGTERM тоже даёт
  `exit 0` — машина не поднимается сама. Фикс — явный блок в `fly.toml`:
  `[[restart]] policy = "always"`.

- **Путь `dockerfile` в `deploy/fly.toml` резолвится относительно директории
  самого `fly.toml`, не относительно корня репозитория.** При явном
  `--config deploy/fly.toml` старое значение `dockerfile = "deploy/Dockerfile"`
  даёт задвоенный путь и падает со сборкой. Верно — `dockerfile = "Dockerfile"`.

- **Память машины (`fly scale memory`) — атрибут конкретной VM, не
  `fly.toml`, не переносится автоматически на новое приложение.** При
  переносе на `anex-analyst-bot` (ANEX-006) новая машина стартовала с
  дефолтными 256MB и тихо зависла тем же образом. Диагностика —
  `fly machine status`, поле `Memory`. При будущих переносах/новых
  приложениях для этого проекта — сразу закладывать 512MB.

- **Fly не поддерживает переименование приложения на месте.** Перенос на
  новое имя требует: создать новое приложение, перенести volume (данные
  скачиваются и загружаются заново через `fly ssh sftp`, не перемещаются
  напрямую), перенести secrets, остановить бота на старом приложении **до**
  запуска нового (иначе два процесса с одним `BOT_TOKEN` дают Telegram
  `409 Conflict`), и только после живой проверки нового — удалить старое.

## Баги и фиксы

Конкретные баги, найденные тестами, с решением — не переоткрывать заново.

- **`_msg_type`: медиа проверяется раньше text.** У Telethon/Bot API `text`
  отдаёт и обычный текст, и подпись к фото/видео — если проверить text
  первым, фото с подписью ошибочно попадёт в "text". При правках в `bot.py`
  и `history.py` сохранять порядок проверки.

- **`asyncio.create_task()` без сохранённой ссылки может быть собран
  сборщиком мусора прямо посреди выполнения.** Воспроизвелось при живом
  тесте `/import_history`: задача подключилась к Telegram и тут же была
  уничтожена. Event loop держит на задачу только слабую ссылку. Фикс в
  `bot.py` — модульный `_background_tasks: set[asyncio.Task]`, задача
  добавляется и удаляется через `task.add_done_callback(...discard)`. Любой
  новый fire-and-forget `asyncio.create_task()` обязан так же делать.

- **Пересинхронизация реакций — снимок за окно, а не «что изменилось».**
  `history._resync_recent_reactions()` пересчитывает реакции у всех
  сообщений за последние `REACTION_RESYNC_LOOKBACK_DAYS` (3) дня — Telegram
  не даёт способа узнать именно дельту. Отсюда цифра в ответе бота может
  быть больше ожидаемой. Сообщение в `bot.py` явно поясняет: «снимок на
  текущий момент» — при правках текста ответа эту оговорку не терять.

- **`st.bar_chart` сортирует строковые категории алфавитно** (дефолт
  Vega-Lite для nominal-полей, не баг версии). Дни недели идут через
  `st.altair_chart` с явным `sort=WEEKDAYS_RU`. Новые графики с именованными
  категориями — сразу проверять сортировку.

## Правила и инварианты

Поведенческие контракты кода, которые нельзя нарушать без пересмотра архитектуры.

- **Сбор идёт только для активированных чатов.** Хендлер сообщений обязан
  начинаться с проверки `is_chat_active(chat_id)` и молча выходить, если чат
  не активирован. Без гейта бот собирает до `/activate`, до второго
  согласия — ломает модель двойного согласия.

- **Дедупликация двух писателей.** В одну БД пишут `bot.py` (live) и
  `history.py` (догрузка назад), их окна пересекаются. Обязателен
  `UNIQUE(chat_id, message_id)` + `INSERT OR IGNORE`.

- **Существующие данные не переклассифицируются задним числом.**
  Авто-миграция меняет схему без удаления `activity.db`, но уже записанные
  строки не переразмечает. Пересчитать классификацию — только осознанным
  передобавлением истории.

- **Авторизация Telethon делается заранее и вручную.** Session-файл
  логинится один раз в консоли (код + 2FA). `/import_history` только
  пользуется готовой сессией. Если сессия невалидна — бот отвечает «нужна
  ре-авторизация в консоли». Интерактивный логин через чат не делаем.

- **Прозрачность для участников.** При `/activate` бот публикует в чате
  сообщение о начале сбора. `/status` доступна любому участнику. Это
  следствие сквозного приоритета приватности, не UX-украшение.

- **Бот выселяют из чата → авто-деактивация.** `my_chat_member`: бота
  удалили/потерял доступ — чат помечается неактивным.

- **`OWNER_USER_ID` — это Telegram user_id, не `TG_API_ID`.** Оба поля —
  числа, легко перепутать при первом деплое. `TG_API_ID` — идентификатор
  Telethon-приложения с my.telegram.org, не связан с личным аккаунтом. Если
  `/chats`/`/import_history` молчат без ошибки в логе — проверять это в
  первую очередь.

- **Импорт истории — фоновая asyncio-задача в процессе бота**, не
  синхронный вызов в хендлере (иначе polling встанет на минуты). Прогресс
  отдаётся в чат по ходу. Импорт идемпотентен.

## Безопасность

Здесь фиксируются принципы (что мы решили и почему) и **журнал инцидентов**
(что реально случилось, как обнаружили, что сделали). Раздел развивается —
каждый новый инцидент добавляется сюда как отдельная запись, не переписывает
предыдущие.

### Принципы

- **В git не попадают секреты и живые данные.** `.env`, `data/` (включая
  `activity.db*` и `history_session.session`) — в `.gitignore`, проверяются
  явно перед каждым первым коммитом/пушем (см. ANEX-007).
- **На проде секреты — только через `fly secrets set`, не в `.env` на
  сервере и не в образе.** `.env` используется только локально.
- **Ключ шифрования в боте — только `DB_ENCRYPTION_KEY`.** Бот —
  долгоживущий процесс без интерактивного stdin, `DB_ENCRYPTION_SALT`
  (пароль) ему недоступен физически.
- **`/import_history`/`/chats` доступны только владельцу** — команда
  задействует его личный Telethon-аккаунт, доступ по `user_id`, не по
  статусу админа.
- **AI Activity Feed: анонимизация текста перед отправкой в Gemini.** Имена/
  юзернеймы заменяются на роль/id (например, `Участник_3`) до формирования
  запроса — контент уходит во внешний API осознанно, но привязка к личности
  наружу уходить не должна. Обязательный шаг пайплайна, не опция.
- **AI Activity Feed: лента и метрики обезличены** — никаких прямых
  указаний на конкретного участника ни в метаданном, ни в текстовом контуре.
- **Секреты никогда не печатаются в вывод инструментов при отладке.**
  Не использовать `grep`/`cat`/`echo` содержимого `.env` или переменных
  окружения в любом выводе, который сохраняется в лог/транскрипт сессии
  (см. инцидент ниже — именно так произошла утечка). Если нужно передать
  переменные в `docker run` — использовать `--env-file .env` напрямую. Если
  нужно проверить, что значение установлено или сгенерировано верно —
  проверять длину/формат/факт наличия (`len(key) == 44` для Fernet-ключа),
  **не печатать само значение**.

### Инциденты

**2026-07 — полный вывод `.env` в транскрипт сессии при отладке ANEX-002/003
(Docker/Fly).**

- **Что произошло.** При диагностике Docker-сборки и памяти на Fly сеньор
  вывел содержимое `.env` целиком (`grep`/аналог) в вывод инструмента —
  все 7 значений (`TG_SESSION`, `TG_PHONE`, `TG_API_ID`, `TG_API_HASH`,
  `BOT_TOKEN`, `OWNER_USER_ID`, `DB_ENCRYPTION_KEY`) осели в транскрипте
  сессии Claude Code. Отдельно фрагмент `BOT_TOKEN` на несколько секунд
  попал в `README.md` как «пример» и был исправлен в том же ходе, до того
  как файл куда-либо ушёл — на тот момент `.git` в проекте ещё не
  существовал (git появился позже, в ANEX-007), риска коммита/пуша не было.
  `activity.db`/`history_session.session` копировались в `/tmp` локально
  для теста контейнера — не в текстовый вывод.

- **Куда реально дошло.** Только транскрипт этой рабочей сессии (локально
  на диске пользователя + обычное хранение истории переписки на стороне
  Anthropic). Наружу, в git, в публичный интернет, во внешние сервисы — не
  ушло. Проверено: `README.md` чист, `/tmp`-копии удалены, `docker history`
  не содержит секретов в слоях образа, `.git` на момент инцидента не
  существовал.

- **Реакция и статус по каждому секрету:**
  - `BOT_TOKEN` — **ротирован** пользователем через `/revoke` в BotFather.
  - `DB_ENCRYPTION_KEY` — **ротирован** сеньором (данные на тот момент были
    тестовые, старые зашифрованные строки осознанно оставлены
    нерасшифровываемыми — не представляли ценности).
  - `TG_API_ID`/`TG_API_HASH` — **не ротированы, осознанно, по решению
    пользователя.** Средний риск (учётные данные Telethon-приложения, сами
    по себе не дают входа в аккаунт без кода 2FA/SMS). Принято как
    допустимый остаточный риск на момент записи — не переоткрывать это
    решение без явного нового обсуждения с архитектором.
  - `TG_PHONE`, `OWNER_USER_ID` — низкий риск, действий не предпринято.
  - `history_session.session` — в текстовый вывод не попадал вообще, файл
    удалён с диска.

- **Извлечённый урок → см. принцип «Секреты никогда не печатаются в вывод
  инструментов» выше.** Это прямое следствие данного инцидента, а не
  абстрактная лучшая практика.

## AI Activity Feed — решения по тону и приватности

- **Тон — нейтральный, профессиональный, не игровой.** Изначальный концепт
  (внешний RFC) предлагал ироничный/развлекательный стиль (названия триггеров
  вроде «Мем-Апокалипсис», метрики вроде «Индекс Душноты»). Решение
  пересмотрено: это серьёзное приложение, работающее с данными реальных
  людей — тон и промпт-инжиниринг в `ai_analyst.py` должны быть нейтральными
  и описательными, не развлекательными. При реализации ANEX (Шаг 1) триггеры
  называть по сути происходящего (например: «всплеск доли медиа-контента»,
  «длинная цепочка обсуждения», «снижение активности»), метрики — по факту,
  что они измеряют, без игровых ярлыков.
- **Два контура безопасности, не смешивать.** Метаданный контур (Шаг 1)
  работает всегда, не имеет доступа к расшифрованному тексту. Текстовый
  контур (Шаг 2) требует активной сессии с паролем, не включается раньше
  фазы 3. `ai_analyst.py` на Шаге 1 не импортирует `crypto.py` — архитектурная
  гарантия, не договорённость.
- **Обезличивание и анонимизация** — см. раздел «Безопасность» выше, это не
  дублирование, обе темы (тон и приватность) исторически обсуждались вместе,
  но это разные решения: тон — про стиль подачи, обезличивание — про защиту
  личности участников.
- **При сбое или лимите Gemini API дашборд показывает последний
  закэшированный результат**, не ошибку и не пустую вкладку.
  `@st.cache_data(ttl="1h")`. AI-метрики — недетерминированные, в UI явно
  помечать как интерпретационные, не как строгую аналитику.
- **`GEMINI_API_KEY` — через тот же секрет-менеджмент, что и остальные
  ключи** (`fly secrets set`). Не в `.env` на проде, не в коде.

## Как проверять готовое, а не просто "не упало"

`py_compile` ловит только синтаксис. Для реальной уверенности:
- Прогонять функции `db.py` напрямую на синтетических данных (не только на
  пустой базе).
- Для `dashboard.py` — поднимать настоящий `streamlit run ... --server.headless
  true --server.port N &`, дожидаться `/_stcore/health`, смотреть лог на
  `Traceback`. Bare `python3 dashboard.py` полезен для логики, но
  `st.stop()` там не останавливает выполнение — это не реальный баг.
- Для `ai_analyst.py` — тестировать с замоканным Gemini-клиентом; отдельно —
  один живой вызов вручную для проверки формата ответа и обработки сбоя.
- Для обёрточного скрипта на Fly — убить один из двух процессов вручную
  (`kill`) и убедиться, что весь контейнер падает и Fly его перезапускает.
- **Перед любым коммитом с изменениями в `.env`-обработке или Docker/Fly-
  конфигах — свериться с принципами раздела «Безопасность» выше.** Не
  печатать секреты в вывод, не запекать их в образ.

## Рабочий процесс

- **Архитектор (Claude.ai Projects, `docs/PROJECT_CONTEXT.md` + этот файл
  через GitHub-синк)** — архитектура, смена парадигмы, выбор между
  вариантами, ревью безопасности. Код здесь не пишем и не тестируем.
- **Сеньор (Claude Code в VS Code)** — вся реализация: правки, тесты,
  запуск. Пишет разделы этого файла напрямую по сложившемуся формату —
  архитектор ревьюит через синк, не пересобирает файл с нуля при каждом
  изменении.
- **Синхронизация — через GitHub-интеграцию Project**, не ручной копипаст.
  Обновление по кнопке «Sync now» в настройках Project — не автоматическое
  в реальном времени.
- Задачи для сеньора — `docs/tasks/ANEX-XXX.md`, сквозная нумерация.
- Находки сеньора о платформенных ограничениях, не видные заранее и
  вскрывшиеся только на реальном деплое — нормальная часть процесса, не
  повод считать план проваленным. Архитектор оценивает предложенный фикс и
  решает, требует ли находка пересмотра архитектуры или это локальная
  коррекция.
- Внешние концепты (например, RFC от другого ИИ-агента) разбираются
  архитектором перед тем, как стать задачей для сеньора — архитектурные,
  приватностные и тональные развилки решает пользователь.

### На будущее (пример, не текущая задача)

Проект может со временем привлекать несколько ИИ-агентов параллельно (не
только Claude Code как сеньора) — например, отдельный агент для
data-science/BI-части поверх той же кодовой базы. Когда это станет
актуальным, потребуется явно зафиксировать: границы по модулям (кто какие
файлы трогает), общий формат задач (возможно, свой аналог `ANEX-XXX.md` для
другого агента), и то, как изменения одного агента синхронизируются с
другим через git, не через прямое взаимодействие агентов друг с другом.
Не реализуется сейчас — просто зарезервировано место в процессе, чтобы не
изобретать это второпях, когда появится реальный второй агент.
