# anex_analyst_bot — контекст проекта

Сбор активности Telegram-чата + Streamlit-дашборд с шифрованием текста.
Инструкция по установке и запуску — в README.md. Стратегические обсуждения
архитектуры — в Claude.ai Projects (`docs/PROJECT_CONTEXT.md`, синхронизируется
через GitHub), с архитектором. Здесь — решения, ограничения, найденные баги,
правила и вопросы безопасности, которые в код и README буквально не попадают,
но важны сеньору при правках.

> Статус: **Фаза 1** и **Фаза 2** реализованы и проверены вживую. Репозиторий
> реорганизован (AAB-5), переименован в `anex_analyst_bot` с переносом
> живого Fly-приложения на `anex-analyst-bot` (AAB-6), выложен на GitHub
> приватно и подключён к Project как источник синхронизации (AAB-7), затем
> сделан **публичным**. После публикации проведён аудит (AAB-8): история
> git чистая (секреты никогда не коммитились), но дашборд обнаружен публично
> доступным без пароля. Первый заход на исправление — **AAB-10** (токен на
> чат целиком + мастер-пароль владельца), почти сразу пересмотрен **AAB-12**:
> токен теперь на пару чат+пользователь (`dashboard_tokens`), доступ к
> дашборду через `/dashboard` доступен любому участнику чата, не только
> админу, плюс `/access_token`/`/revoke_token` для точечной выдачи/отзыва
> владельцем. Мастер-пароль (`DASHBOARD_PASSWORD`) не тронут. Риск AAB-8
> закрыт — см. журнал инцидентов в разделе «Безопасность». **AI Activity
> Feed** — концепт принят, Шаг 1 (метаданные) готов к реализации, не начат.
> Тон фичи — нейтральный/профессиональный, не игровой (решение
> зафиксировано ниже, в разделе AI Activity Feed).

## Структура репозитория

```
anex_analyst_bot/
├── CLAUDE.md              # только в корне — Claude Code читает автоматически отсюда
├── README.md, .env, .env.example, .gitignore, .dockerignore, requirements.txt
├── data/                  # activity.db + Telethon session, gitignored, НЕ в src/
├── deploy/                # Dockerfile, fly.toml, start.sh
├── docs/
│   ├── PROJECT_CONTEXT.md # стратегический документ для архитектора
│   └── tasks/             # AAB-N.md — задачи для сеньора
└── src/                   # bot.py, crypto.py, dashboard.py, db.py, generate_key.py, history.py
```

- **Задачи для сеньора — `docs/tasks/AAB-N.md`, без ведущих нулей** (не
  `AAB-001`, а `AAB-1`). До AAB-11 формат был `ANEX-0XX.md` с ведущими нулями
  — сменён, так как фиксированная ширина в 3 цифры упирается в потолок на
  999-й задаче. Сквозная нумерация, без пропусков и без переиспользования
  номеров, как и раньше: AAB-1 = фаза 1 (бот), AAB-2 = фаза 2 (Fly.io),
  AAB-3 = фаза 2-коррекция (одна машина), AAB-4 = AI Activity Feed Шаг 1,
  AAB-5 = реорганизация папок, AAB-6 = унификация имени, AAB-7 = git init +
  push на GitHub, AAB-8 = аудит после публикации репозитория, AAB-9 = задача
  архитектора, AAB-10 = авторизация дашборда (токен на чат + мастер-пароль),
  AAB-11 = смена формата номеров + новые правила процесса (подтверждение
  закрытия задачи, проверка на утечку секретов перед push — см. «Рабочий
  процесс» и «Безопасность» ниже), AAB-12 = пересмотр модели токена AAB-10
  на пару чат+пользователь, команды `/dashboard`/`/access_token`/
  `/revoke_token`. Далее — та же сквозная нумерация в этом же формате.
- **Модули запускаются как `python src/bot.py` и т.п. из корня репозитория**,
  не из `cd src`. Соседние импорты (`import db`, `from crypto import ...`)
  работают без изменений — Python кладёт директорию самого скрипта в
  `sys.path`, переход на пакетные импорты (`src/__init__.py`) не нужен.
- **Путь к `activity.db`/session — параметризован (`DATA_DIR` или аналог),
  не хардкод.** Локальный дефолт — `data/`. Значение на Fly (смонтированный
  volume) реорганизацией репозитория не меняется — это два независимых слоя.

## Архитектура (кто за что отвечает)

- `db.py` — схема SQLite + вся SQL-аналитика, единая точка правды, от неё
  зависят все модули. `msg_type` по умолчанию `"text"` везде, кроме реакций
  и разбивки по типам. Авто-миграция схемы (не нужно удалять `activity.db`
  при изменении схемы). AI Activity Feed добавляет лёгкие агрегирующие
  запросы для аномалий флуда, цепочек реплаев, реактивности и матрицы день×час.
- `crypto.py` — шифрование `text_encrypted` (Fernet), два режима ключа.
  Модуль, который пишет данные, обязан иметь ключ — без него запись
  физически отклоняется.
- `bot.py` — aiogram, Bot API. Постоянно живёт в чате, основной механизм
  сбора. Выбор чата динамический через команды. В группе: `/activate`,
  `/deactivate`, `/status` (админ чата или владелец, кроме `/status` — тот
  доступен всем). В личке с ботом, любой пользователь: `/dashboard` (AAB-12
  — своя ссылка на дашборд чата, где участвуешь), `/help` (AAB-16 — список
  команд по роли: владелец/админ активного чата/остальные, см. «Правила и
  инварианты»). В личке, только владелец: `/chats`, `/import_history`,
  `/access_token`, `/revoke_token` (AAB-12). Сборщик (`collect_message`)
  сохраняет только allowlist реального контента участника — служебные
  сообщения Telegram и собственные исходящие бота отфильтровываются до
  записи в БД (AAB-16, см. «Правила и инварианты» и «Баги и фиксы»). Живой
  трекинг реакций (AAB-17) — `on_message_reaction`/`on_message_reaction_count`,
  **требует, чтобы бот был админом в чате** (платформенное ограничение, см.
  «Платформенные ограничения» ниже), иначе апдейты не приходят вообще.
- `history.py` — точечная догрузка истории личным аккаунтом (Telethon),
  подключается только на время команды `/import_history`, не висит
  постоянным процессом. Заодно пересчитывает реакции за последние 3 дня
  (актуально только для сообщений, которые ещё не покрыты живым трекингом
  AAB-17 — например, импортированных задним числом).
- `dashboard.py` — Streamlit. Только читает БД, никогда не пишет (кроме
  вызовов Gemini API из `ai_analyst.py`). Вкладки Обзор/Участники/Сообщения/
  Текст, тепловая карта день×час, тренд по дням. AI Activity Feed — новая
  вкладка «AI Инсайды».
- `ai_analyst.py` — новый (план), интеграция с `google-genai`. Изолированный
  модуль. Принимает структурированные метаданные из `db.py`, отдаёт
  типизированный JSON для карточек ленты. Не имеет доступа к `crypto.py` на
  Шаге 1.
- `generate_key.py` — утилита генерации ключа шифрования.
- Хостинг — Fly.io, одна машина на оба процесса (`bot.py`+`history.py` по
  требованию и `dashboard.py`), обёрточный скрипт `start.sh`. Машина —
  `shared-cpu-1x:512MB`. Приложение — `anex-analyst-bot`.

## Решения

Осознанные архитектурные выборы и почему не выбраны альтернативы.

- **Bot API — основной сбор, личный аккаунт — только для истории.**
  Постоянный сбор идёт через бота — паттерн, под который Telegram
  проектировал автоматизацию, без вопросов к безопасности. Личный аккаунт
  (Telethon) подключается разово по `/import_history`. Bot API принципиально
  не читает историю до момента добавления бота — платформенная граница, для
  истории нужен `history.py`.

- **Шифруется только поле text, не вся БД.** SQLCipher рассматривали и
  отклонили: системная C-библиотека, хрупкая установка на скудном окружении.
  Метаданные (кто/когда/тип/реакции) открыты — на них строится вся аналитика.

- **Два режима ключа шифрования**: `DB_ENCRYPTION_KEY` (статический, в `.env`
  локально / `fly secrets` на проде) или `DB_ENCRYPTION_SALT` (пароль,
  только для дашборда и консольных запусков — боту недоступен, он
  долгоживущий процесс без интерактивного stdin).

- **Живой трекинг реакций через message_reaction/message_reaction_count, не
  расширение окна пересинхронизации (AAB-17).** По итогам разбора AAB-17
  выяснилось, что до этой задачи реакции не обновлялись «на лету» вообще —
  единственный источник свежих данных был ручной `/import_history`, а
  задокументированное «окно в 3 дня» было на деле более узким частным
  случаем. Решение архитектора — писать реакции живыми Bot API апдейтами
  (требуют, чтобы бот был админом чата, см. «Платформенные ограничения»),
  а не просто увеличивать `REACTION_RESYNC_LOOKBACK_DAYS`. `message_reaction`
  сообщает изменение только одного автора за раз (старый набор реакций →
  новый), не общий счётчик сообщения — поэтому состояние по каждому автору
  накапливается в `message_reactions_by_actor` (`actor_id` — `user_id` или
  id анонимного `actor_chat`, пространства не пересекаются: `user_id`
  положительный, `chat_id` всегда отрицательный), а `reactions.reaction_count`
  пересчитывается как сумма при каждом апдейте. `message_reaction_count`
  (агрегированный, анонимные реакции) просто перезаписывает total целиком.
  `history.py`/резюнк на 3 дня не убраны — они покрывают сообщения,
  импортированные задним числом, куда live-апдейты в принципе не доходят.

- **Хостинг — Fly.io, не VPS.** Два долгоживущих процесса и два файла с
  нужным персистентным диском (`activity.db`, Telethon session) — отсекает
  PaaS с эфемерной файловой системой. Fly даёт persistent volumes и secrets
  без администрирования ОС — оправдано профилем пользователя (аналитик, не
  системный администратор).

- **Одна Fly Machine на оба процесса, не два `[processes]`.** См. «Платформенные
  ограничения» — исходный план не сработал технически, не по выбору.

- **`/deactivate` только останавливает сбор, данные (`messages`/`reactions`)
  не трогает.** Удаление — отдельной явной командой при необходимости.
  Случайный `/deactivate` не должен стирать историю необратимо. **Токены
  дашборда — отдельно от данных, их `/deactivate` как раз отзывает** (см.
  инвариант про `dashboard_tokens` ниже) — это не противоречие, а разные
  сущности: собранная история и текущий доступ к её просмотру.

- **`/import_history` и `/chats` — только владелец деплоя** (`OWNER_USER_ID`).
  Задействует личный Telethon-аккаунт владельца — админ стороннего чата не
  должен иметь возможность это инициировать.

- **`/activate`/`/deactivate` — админ чата ИЛИ владелец, не только админ.**
  Владелец управляет сбором в любом чате, даже не будучи там админом.
  Отдельно от правила выше: `/import_history`/`/chats` — строго только
  владелец, без варианта «или админ».

- **Унификация имени: `anex_analyst_bot` vs `anex-analyst-bot`.** Бот в
  Telegram уже назывался `anex_analyst_bot` (Telegram не разрешает дефис в
  username). Fly.io не разрешает подчёркивание в имени приложения — значит
  там `anex-analyst-bot`. Единственное осознанное расхождение, не путать со
  старой случайной нестыковкой `tg_chat_analytics`/`tg-chat-analytics`.

## Платформенные ограничения

Факты о Telegram/Fly/Streamlit, которые мы не выбирали, а обязаны учитывать.

- **Privacy Mode в BotFather обязан быть выключен.** По умолчанию Bot API
  отдаёт боту только команды и ответы/упоминания — обычные сообщения чата не
  приходят вообще. Нужно `/setprivacy → Disable` **и передобавить бота в
  чат** (смена применяется только при повторном входе). Симптом, если
  забыли: бот «работает», ошибок нет, но в БД почти пусто.

- **`message_reaction`/`message_reaction_count` доставляются боту, только
  если бот — админ чата.** Без ошибки, без исключения — апдейты просто не
  приходят вообще, тот же тихий симптом, что у Privacy Mode выше. Живой
  трекинг реакций (AAB-17) требует **вручную повысить бота до админа в
  каждом чате**, где он собирает статистику — это отдельное действие,
  `/activate` его не делает и не проверяет. Если после AAB-17 реакции
  в чате всё равно не обновляются live — начинать проверку именно отсюда.

- **MTProto блокируется у пользователя на уровне провайдера.** VPN и
  публичные MTProxy не сработали надёжно. Рабочее решение —
  `system_version="4.16.30-vxCUSTOM"` при создании `TelegramClient` в
  `history.py`. При проблемах с подключением личного аккаунта — начинать с
  проверки этого параметра, а не городить прокси заново.

- **`chat_id` у Telethon и Bot API разный** для супергрупп/каналов (голый id
  vs `-100`-префикс). В одну БД пишут два источника — `bot.py` (Bot API) и
  `history.py` (Telethon), везде обязателен `telethon.utils.get_peer_id()`.

- **Volume на Fly.io физически привязан к одной машине — не шарится между
  несколькими.** Исходный план «`bot` и `dashboard` как два отдельных
  `[processes]`» не работает: Fly создаёт под каждый process group отдельную
  машину и отдельный volume. Исправление: оба процесса — внутри одной Fly
  Machine через обёрточный скрипт.

- **Вся машина не может автостопиться, пока жив `bot`.** `bot` использует
  long polling и не должен засыпать никогда — значит вся машина держится
  постоянно включённой (`auto_stop_machines = false`).

- **Дефолтные 256MB памяти на Fly Machine не хватает на bot+dashboard
  вместе.** ~90MB RSS у бота и ~40MB у дашборда сразу после старта — `MemFree`
  падает до ~2.7MB, машина тихо виснет (не отвечает ни health-check, ни SSH),
  без понятной ошибки. Фикс — `flyctl scale memory 512`. При добавлении
  ещё одного тяжёлого компонента в контейнер (например, `ai_analyst.py`) —
  сразу закладывать память с запасом.

- **Fly не перезапускает машину при чистом `exit 0` без явной restart-политики.**
  Дефолт — `restart.policy = "on-failure"`: код выхода 0 трактуется как
  «намеренная остановка». Fail-fast скрипт при штатном SIGTERM тоже даёт
  `exit 0` — машина не поднимается сама. Фикс — явный блок в `fly.toml`:
  `[[restart]] policy = "always"`.

- **Путь `dockerfile` в `deploy/fly.toml` резолвится относительно директории
  самого `fly.toml`, не относительно корня репозитория.** При явном
  `--config deploy/fly.toml` старое значение `dockerfile = "deploy/Dockerfile"`
  даёт задвоенный путь и падает со сборкой. Верно — `dockerfile = "Dockerfile"`.

- **Память машины (`fly scale memory`) — атрибут конкретной VM, не
  `fly.toml`, не переносится автоматически на новое приложение.** При
  переносе на `anex-analyst-bot` (AAB-6) новая машина стартовала с
  дефолтными 256MB и тихо зависла тем же образом. Диагностика —
  `fly machine status`, поле `Memory`. При будущих переносах/новых
  приложениях для этого проекта — сразу закладывать 512MB.

- **Fly не поддерживает переименование приложения на месте.** Перенос на
  новое имя требует: создать новое приложение, перенести volume (данные
  скачиваются и загружаются заново через `fly ssh sftp`, не перемещаются
  напрямую), перенести secrets, остановить бота на старом приложении **до**
  запуска нового (иначе два процесса с одним `BOT_TOKEN` дают Telegram
  `409 Conflict`), и только после живой проверки нового — удалить старое.

- **`fly deploy` разворачивает содержимое локальной рабочей директории на
  момент запуска команды — не состояние git (ни закоммиченное, ни
  запушенное).** Значит: закоммиченный и запушенный код не обязательно уже
  работает на проде, пока явно не выполнен `fly deploy`; и наоборот — можно
  случайно задеплоить незакоммиченные изменения. Обнаружено при AAB-12: фикс
  не проявлялся на проде именно по этой причине. После любой задачи с
  изменениями в `src/` или `deploy/`, которые должны попасть на прод — явно
  выполнять `fly deploy` и не считать факт коммита/пуша признаком того, что
  изменения уже живые.

## Баги и фиксы

Конкретные баги, найденные тестами, с решением — не переоткрывать заново.

- **`_msg_type`: медиа проверяется раньше text.** У Telethon/Bot API `text`
  отдаёт и обычный текст, и подпись к фото/видео — если проверить text
  первым, фото с подписью ошибочно попадёт в "text". При правках в `bot.py`
  и `history.py` сохранять порядок проверки.

- **`asyncio.create_task()` без сохранённой ссылки может быть собран
  сборщиком мусора прямо посреди выполнения.** Воспроизвелось при живом
  тесте `/import_history`: задача подключилась к Telegram и тут же была
  уничтожена. Event loop держит на задачу только слабую ссылку. Фикс в
  `bot.py` — модульный `_background_tasks: set[asyncio.Task]`, задача
  добавляется и удаляется через `task.add_done_callback(...discard)`. Любой
  новый fire-and-forget `asyncio.create_task()` обязан так же делать.

- **Пересинхронизация реакций — снимок за окно, а не «что изменилось».**
  `history._resync_recent_reactions()` пересчитывает реакции у всех
  сообщений за последние `REACTION_RESYNC_LOOKBACK_DAYS` (3) дня — Telegram
  не даёт способа узнать именно дельту. Отсюда цифра в ответе бота может
  быть больше ожидаемой. Сообщение в `bot.py` явно поясняет: «снимок на
  текущий момент» — при правках текста ответа эту оговорку не терять.
  **Уточнение по итогам AAB-17:** до AAB-17 это была не просто «окно в 3
  дня» — реакции вообще нигде не обновлялись, кроме как во время ручного
  `/import_history`; свежесть данных дашборда целиком зависела от того,
  когда владелец последний раз запускал команду. Живой трекинг закрывает
  именно это (см. запись в «Решения»); резюнк на 3 дня остаётся отдельно —
  для сообщений, импортированных задним числом, куда live-трекинг не
  дотягивается (аккаунт-history не проходит через `bot.py`).

- **Служебные сообщения Telegram и собственные сообщения бота попадали в
  `messages` как активность участника (AAB-16).** До фикса `_msg_type`
  классифицировал любой неизвестный `content_type` как `"other"` вместо
  того, чтобы отбросить сообщение — реальные симптомы, найденные на живых
  данных чата: (1) 4 строки `msg_type="other"` на реального участника —
  служебные события (пин/смена названия и т.п.), раздувавшие его счётчик
  активности; (2) 4 строки `msg_type="text"`, `user_id`/`username` = сам бот
  (`anex_analyst_bot`) — собственные ответы бота в группе (`message.reply`/
  `message.answer`) реально доставляются ему обратно как `message`-апдейты,
  вопреки ожиданию, что бот не получает свои же исходящие сообщения. Из-за
  этого бот попадал в список участников чата (`db.chat_participant_user_ids`,
  используется `/access_token` без `user_id`) и `/access_token` пытался
  выдать ему токен, а рассылка падала с «нет диалога в личке с ботом» на
  user_id самого бота. Фикс — allowlist content_type + явное исключение
  `user.id == bot_id` в `_save_message` (см. инвариант выше). Уже
  накопленные до фикса 8 строк (4 служебных + 4 бота, в `data/activity.db`
  локально на момент находки, чат «Тест группа») **не удалялись** — решение
  об очистке отложено на архитектора/пользователя, см. процесс задачи.

- **`st.bar_chart` сортирует строковые категории алфавитно** (дефолт
  Vega-Lite для nominal-полей, не баг версии). Дни недели идут через
  `st.altair_chart` с явным `sort=WEEKDAYS_RU`. Новые графики с именованными
  категориями — сразу проверять сортировку.

## Правила и инварианты

Поведенческие контракты кода, которые нельзя нарушать без пересмотра архитектуры.

- **Сбор идёт только для активированных чатов.** Хендлер сообщений обязан
  начинаться с проверки `is_chat_active(chat_id)` и молча выходить, если чат
  не активирован. Без гейта бот собирает до `/activate`, до второго
  согласия — ломает модель двойного согласия.

- **Дедупликация двух писателей.** В одну БД пишут `bot.py` (live) и
  `history.py` (догрузка назад), их окна пересекаются. Обязателен
  `UNIQUE(chat_id, message_id)` + `INSERT OR IGNORE`.

- **Существующие данные не переклассифицируются задним числом.**
  Авто-миграция меняет схему без удаления `activity.db`, но уже записанные
  строки не переразмечает. Пересчитать классификацию — только осознанным
  передобавлением истории.

- **Авторизация Telethon делается заранее и вручную.** Session-файл
  логинится один раз в консоли (код + 2FA). `/import_history` только
  пользуется готовой сессией. Если сессия невалидна — бот отвечает «нужна
  ре-авторизация в консоли». Интерактивный логин через чат не делаем.

- **Прозрачность для участников.** При `/activate` бот публикует в чате
  сообщение о начале сбора. `/status` доступна любому участнику. Это
  следствие сквозного приоритета приватности, не UX-украшение. Расширено в
  AAB-15: при появлении **новых** участников в уже активном чате
  (`new_chat_members`) бот публикует туда же короткое приветствие с
  напоминанием про `/status` и `/dashboard` — иначе новый участник не мог бы
  узнать об этих командах никаким другим способом (проактивно написать ему
  в личку бот не может, Telegram не разрешает первым). В неактивном чате
  сообщение не публикуется — гейт тот же `is_chat_active`, что и у сбора.
  Добавление самого бота в чат этим обработчиком не триггерится — это
  отдельный флоу `/activate` через `on_my_chat_member`.

- **Бот выселяют из чата → авто-деактивация.** `my_chat_member`: бота
  удалили/потерял доступ — чат помечается неактивным.

- **`OWNER_USER_ID` — это Telegram user_id, не `TG_API_ID`.** Оба поля —
  числа, легко перепутать при первом деплое. `TG_API_ID` — идентификатор
  Telethon-приложения с my.telegram.org, не связан с личным аккаунтом. Если
  `/chats`/`/import_history` молчат без ошибки в логе — проверять это в
  первую очередь.

- **В `messages` попадает только allowlist реального контента участника
  (AAB-16).** `_ALLOWED_CONTENT_TYPES` в `bot.py` = `_DIRECT_TYPES ∪
  _OTHER_MEDIA_TYPES` (text/photo/sticker/video/voice/document/audio/
  video_note/animation). Всё остальное — служебные content_type Telegram
  (`left_chat_member`, `pinned_message`, `new_chat_title`, `new_chat_photo`,
  `migrate_to_chat_id` и т.п., плюс `new_chat_members` помимо явно
  обработанного в `on_new_chat_members`) — `_save_message` отбрасывает
  молча, до записи в БД. Раньше любой неизвестный content_type всё равно
  сохранялся с `msg_type="other"` — см. инцидент в «Баги и фиксы».
  Собственные исходящие сообщения бота в группу тоже отбрасываются
  (`user.id == bot_id`) — подтверждено на реальных данных, что Bot API
  доставляет их бэком как обычные `message`-апдейты, не только чужие.

- **Импорт истории — фоновая asyncio-задача в процессе бота**, не
  синхронный вызов в хендлере (иначе polling встанет на минуты). Прогресс
  отдаётся в чат по ходу. Импорт идемпотентен.

- **Доступ к дашборду одного чата не даёт доступа к данным других чатов
  (AAB-10, модель токена пересмотрена в AAB-12).** Токен — на пару
  `(chat_id, user_id)` в таблице `dashboard_tokens`, не на чат целиком:
  каждый участник получает свою ссылку, владелец может отозвать доступ
  одному человеку, не трогая остальных (`/revoke_token`). Персональная
  ссылка несёт только `?token=<t>`, без `chat_id`/`user_id` в URL (решение
  ещё из AAB-10 — светить идентификаторы в адресной строке/логах незачем,
  сам по себе такой идентификатор не секрет и не даёт доступа). Чат
  находится обратным поиском по самому токену,
  `db.get_chat_id_by_dashboard_token()`, каждая строка сравнивается
  `hmac.compare_digest`. При успехе `dashboard.py` обязан фильтровать все
  вкладки строго по найденному `chat_id` и не показывать селектор других
  чатов в этом режиме. Только `DASHBOARD_PASSWORD` (мастер-пароль
  владельца) даёт вид со всеми чатами сразу — и имеет приоритет, если
  открыты оба способа входа одновременно.

- **`/activate` и `/dashboard` создают токен лениво и идемпотентно, не
  перевыпускают его на каждый вызов (AAB-12).** Повторный `/activate` —
  например, другим админом того же чата — не должен молча убивать уже
  выданную и используемую ссылку первого админа (вместо этого повторный
  `/activate` на уже активном чате отвечает «уже запущен», см. отдельный
  инвариант выше про сам факт активации). Перевыпуск токена — только явное
  действие того, кому он принадлежит: повторный вызов `/dashboard` самим
  этим пользователем. `/access_token` (владелец) при массовой выдаче тоже не
  перевыпускает уже существующие токены участников — иначе повторный запуск
  команды владельцем сломал бы всем уже разосланные ссылки.

- **`/deactivate` отзывает все токены дашборда этого чата и уведомляет
  каждого затронутого пользователя в личку (AAB-12, добавлено сверх
  исходной задачи).** Отклонение от изначального AAB-12 («на токены не
  влияет») — по прямому запросу пользователя: даёт способ быстро закрыть
  доступ к дашборду чата одной командой в критичной ситуации (например,
  подозрение на утечку ссылки), не вызывая `/revoke_token` для каждого
  участника отдельно. Повторная активация (`/activate`) после этого выдаёт
  свежий токен, как при первом включении — старый восстановить нельзя.

## Безопасность

Здесь фиксируются принципы (что мы решили и почему) и **журнал инцидентов**
(что реально случилось, как обнаружили, что сделали). Раздел развивается —
каждый новый инцидент добавляется сюда как отдельная запись, не переписывает
предыдущие.

### Принципы

- **Персональная ссылка на дашборд — предъявительский доступ (bearer token),
  не привязана к личности технически.** `dashboard.py` проверяет только сам
  токен в URL (`hmac.compare_digest` по строкам `dashboard_tokens`) — у него
  физически нет способа узнать, кто именно открыл ссылку в браузере
  (Telegram Login Widget осознанно не строили, см. AAB-10/12). Значит: у
  кого есть точная строка `?token=`, тот и видит дашборд чата, независимо от
  того, кто он на самом деле. Это осознанное следствие выбранной модели
  (простая персональная ссылка вместо полноценного логина через Telegram), а
  не ошибка в коде (подтверждено при разборе AAB-14 — выдача и поиск токена
  корректно скопированы на пару `(chat_id, user_id)`, коллизий нет). Ссылку
  нельзя пересылать и нельзя открывать не тому, кому она предназначена.
  Утечка или намеренный шеринг = чужой доступ к данным чата; лечится
  `/revoke_token` владельцем (точечно) и перевыпуском через `/dashboard`
  самим пострадавшим пользователем.
- **В git не попадают секреты и живые данные.** `.env`, `data/` (включая
  `activity.db*` и `history_session.session`) — в `.gitignore`, проверяются
  явно перед каждым первым коммитом/пушем (см. AAB-7).
- **На проде секреты — только через `fly secrets set`, не в `.env` на
  сервере и не в образе.** `.env` используется только локально.
- **Ключ шифрования в боте — только `DB_ENCRYPTION_KEY`.** Бот —
  долгоживущий процесс без интерактивного stdin, `DB_ENCRYPTION_SALT`
  (пароль) ему недоступен физически.
- **`/import_history`/`/chats` доступны только владельцу** — команда
  задействует его личный Telethon-аккаунт, доступ по `user_id`, не по
  статусу админа.
- **AI Activity Feed: анонимизация текста перед отправкой в Gemini.** Имена/
  юзернеймы заменяются на роль/id (например, `Участник_3`) до формирования
  запроса — контент уходит во внешний API осознанно, но привязка к личности
  наружу уходить не должна. Обязательный шаг пайплайна, не опция.
- **AI Activity Feed: лента и метрики обезличены** — никаких прямых
  указаний на конкретного участника ни в метаданном, ни в текстовом контуре.
- **Секреты никогда не печатаются в вывод инструментов при отладке.**
  Не использовать `grep`/`cat`/`echo` содержимого `.env` или переменных
  окружения в любом выводе, который сохраняется в лог/транскрипт сессии
  (см. инцидент ниже — именно так произошла утечка). Если нужно передать
  переменные в `docker run` — использовать `--env-file .env` напрямую. Если
  нужно проверить, что значение установлено или сгенерировано верно —
  проверять длину/формат/факт наличия (`len(key) == 44` для Fernet-ключа),
  **не печатать само значение**.
- **Обязательная проверка на утечку секретов перед любым `git push` (правило
  действует начиная с AAB-11, для всех задач, не только про безопасность).**
  Логическое продолжение принципа выше — не только не печатать секреты в
  вывод, но и явно перепроверять перед каждой выкладкой: (1) `git diff
  --staged` (или `git show` на ещё не запушенные коммиты) визуально
  проверяется на предмет значений секретов из `.env` (`TG_SESSION`,
  `TG_PHONE`, `TG_API_ID`, `TG_API_HASH`, `BOT_TOKEN`, `OWNER_USER_ID`,
  `DB_ENCRYPTION_KEY`, `DASHBOARD_PASSWORD`, `GEMINI_API_KEY` при появлении)
  — именно значения, не сами имена переменных; (2) если секреты печатались в
  собственном рабочем выводе инструментов за время задачи (терминал, логи
  отладки) — это не блокирует коммит сам по себе, но должно быть явно
  отмечено в отчёте пользователю, не замалчиваться; (3) если что-то найдено —
  не пушить, сообщить архитектору через пользователя, историю самостоятельно
  не чистить, ждать решения; (4) это часть обычного цикла для каждой задачи,
  не отдельный чек-лист, который можно пропустить для «некритичных» правок.

**2026-07-21 — публичный доступ к дашборду закрыт, добавлена модель с
разделением по чатам (AAB-10).**

- **Что сделано.** Два независимых способа входа вместо полного отсутствия
  аутентификации: (1) персональная ссылка `?token=<t>` — токен
  (`secrets.token_urlsafe(32)`) в новой колонке `chats.chat_access_token`,
  выдаётся ботом в личку по `/activate`/`/dashboard_link`, чат находится
  обратным поиском по токену (`db.get_chat_id_by_token()`, каждая строка
  сравнивается `hmac.compare_digest`), открывает только один `chat_id`; (2)
  `DASHBOARD_PASSWORD` — мастер-пароль владельца через `fly secrets set`,
  открывает вид со всеми чатами и селектором, как раньше. Без одного из
  двух — дашборд не показывает вообще ничего, включая метаданные. Изначально
  ссылка была `?chat=<id>&token=<t>` — по итогам живого теста `chat_id`
  убрали из URL: сам по себе не секрет и не даёт доступа без токена, но
  незачем светить его лишний раз в адресной строке/логах.
- **Почему не общий пароль (первоначальный план).** `/activate`/`/deactivate`
  доступны админу любого чата, не только владельцу — общий пароль на весь
  дашборд означал бы, что такой админ видит и чужие чаты. См. новый
  инвариант в разделе «Правила и инварианты».
- **Период открытого риска.** С момента публикации репозитория (аудит
  AAB-8) по 2026-07-21 — дашборд отдавал метаданные активности всех чатов
  без какой-либо аутентификации любому, кто узнал URL.
- **Модель токена пересмотрена в тот же день, см. запись ниже (AAB-12)** —
  токен на чат целиком (`chats.chat_access_token`) заменён на токен пары
  чат+пользователь, `/dashboard_link` заменена на `/dashboard`.

**2026-07-21 — модель токена доступа пересмотрена: с чата целиком на пару
чат+пользователь (AAB-12).**

- **Почему пересмотрели в тот же день.** По итогам обсуждения с
  пользователем — `/activate` в модели AAB-10 совмещала две разные
  ответственности (включить сбор + выдать секретный доступ к дашборду), а
  отдельная команда `/dashboard_link` создавала лишнюю когнитивную нагрузку
  (нужно было помнить о её существовании). Решение вынесено архитектору, не
  придумано на месте сеньором — см. `docs/tasks/AAB-12.md`.
- **Что изменилось.** Токен теперь в таблице `dashboard_tokens`, на пару
  `(chat_id, user_id)`, не на чат целиком (`chats.chat_access_token` из
  AAB-10 остаётся в схеме неиспользуемой — миграции не переписываются
  задним числом). Доступ к дашборду теперь получает **любой участник чата**
  через `/dashboard` в личке с ботом (не только админ) — первый вызов
  выдаёт ссылку, повторный перевыпускает (старая сразу умирает). `/activate`
  по-прежнему остаётся идемпотентным — не перевыпускает токен на повторный
  вызов, только выдаёт свой первым вызовом. Добавлены `/access_token`
  (владелец выдаёт доступ любому участнику вручную) и `/revoke_token`
  (владелец отзывает — точечно одному человеку или всем сразу по чату).
  `/dashboard_link` удалена, её функцию полностью взяла на себя `/dashboard`.
- **Мастер-пароль не тронут** — независимый способ входа, как и был.
- **Доработки по итогам живого теста, тем же днём:** (1) повторный
  `/activate` на уже активном чате отвечает «уже запущен... /deactivate»,
  а не молча переотправляет то же «Запущен сбор...», как будто только что
  включили — иначе путало пользователя; (2) `/deactivate` теперь отзывает
  все токены дашборда чата и уведомляет каждого затронутого — отклонение от
  исходного текста задачи («на токены не влияет»), добавлено по прямому
  запросу пользователя ради быстрого контроля в критичной ситуации. Данные
  `messages`/`reactions` оба раза не затронуты.

- **Тон — нейтральный, профессиональный, не игровой.** Изначальный концепт
  (внешний RFC) предлагал ироничный/развлекательный стиль (названия триггеров
  вроде «Мем-Апокалипсис», метрики вроде «Индекс Душноты»). Решение
  пересмотрено: это серьёзное приложение, работающее с данными реальных
  людей — тон и промпт-инжиниринг в `ai_analyst.py` должны быть нейтральными
  и описательными, не развлекательными. При реализации ANEX (Шаг 1) триггеры
  называть по сути происходящего (например: «всплеск доли медиа-контента»,
  «длинная цепочка обсуждения», «снижение активности»), метрики — по факту,
  что они измеряют, без игровых ярлыков.
- **Два контура безопасности, не смешивать.** Метаданный контур (Шаг 1)
  работает всегда, не имеет доступа к расшифрованному тексту. Текстовый
  контур (Шаг 2) требует активной сессии с паролем, не включается раньше
  фазы 3. `ai_analyst.py` на Шаге 1 не импортирует `crypto.py` — архитектурная
  гарантия, не договорённость.
- **Обезличивание и анонимизация** — см. раздел «Безопасность» выше, это не
  дублирование, обе темы (тон и приватность) исторически обсуждались вместе,
  но это разные решения: тон — про стиль подачи, обезличивание — про защиту
  личности участников.
- **При сбое или лимите Gemini API дашборд показывает последний
  закэшированный результат**, не ошибку и не пустую вкладку.
  `@st.cache_data(ttl="1h")`. AI-метрики — недетерминированные, в UI явно
  помечать как интерпретационные, не как строгую аналитику.
- **`GEMINI_API_KEY` — через тот же секрет-менеджмент, что и остальные
  ключи** (`fly secrets set`). Не в `.env` на проде, не в коде.

## Как проверять готовое, а не просто "не упало"

`py_compile` ловит только синтаксис. Для реальной уверенности:
- Прогонять функции `db.py` напрямую на синтетических данных (не только на
  пустой базе).
- Для `dashboard.py` — поднимать настоящий `streamlit run ... --server.headless
  true --server.port N &`, дожидаться `/_stcore/health`, смотреть лог на
  `Traceback`. Bare `python3 dashboard.py` полезен для логики, но
  `st.stop()` там не останавливает выполнение — это не реальный баг.
- Для `ai_analyst.py` — тестировать с замоканным Gemini-клиентом; отдельно —
  один живой вызов вручную для проверки формата ответа и обработки сбоя.
- Для обёрточного скрипта на Fly — убить один из двух процессов вручную
  (`kill`) и убедиться, что весь контейнер падает и Fly его перезапускает.
- **Перед любым коммитом с изменениями в `.env`-обработке или Docker/Fly-
  конфигах — свериться с принципами раздела «Безопасность» выше.** Не
  печатать секреты в вывод, не запекать их в образ.

## Рабочий процесс

- **Архитектор (Claude.ai Projects, `docs/PROJECT_CONTEXT.md` + этот файл
  через GitHub-синк)** — архитектура, смена парадигмы, выбор между
  вариантами, ревью безопасности. Код здесь не пишем и не тестируем.
- **Сеньор (Claude Code в VS Code)** — вся реализация: правки, тесты,
  запуск. Пишет разделы этого файла напрямую по сложившемуся формату —
  архитектор ревьюит через синк, не пересобирает файл с нуля при каждом
  изменении.
- **Синхронизация — через GitHub-интеграцию Project**, не ручной копипаст.
  Обновление по кнопке «Sync now» в настройках Project — не автоматическое
  в реальном времени.
- Задачи для сеньора — `docs/tasks/AAB-N.md`, сквозная нумерация, без ведущих нулей.
- Находки сеньора о платформенных ограничениях, не видные заранее и
  вскрывшиеся только на реальном деплое — нормальная часть процесса, не
  повод считать план проваленным. Архитектор оценивает предложенный фикс и
  решает, требует ли находка пересмотра архитектуры или это локальная
  коррекция.
- Внешние концепты (например, RFC от другого ИИ-агента) разбираются
  архитектором перед тем, как стать задачей для сеньора — архитектурные,
  приватностные и тональные развилки решает пользователь.
- **Подтверждение перед закрытием задачи (правило действует начиная с AAB-11).**
  Сеньор не закрывает задачу по своему усмотрению. Порядок: (1) довести
  работу до состояния «критерии готовности выполнены»; (2) явно спросить
  пользователя в чате «Задачу [AAB-N] закрываем?» и дождаться ответа; (3)
  только после подтверждения — заполнить итоговый контекст в файле задачи
  (что сделано, что нашли, отклонения от плана, по аналогии с AAB-8) и
  закоммитить это вместе с остальными изменениями задачи; (4) если
  пользователь просит доработать — не закрывать, вернуться к пункту 1.

### На будущее (пример, не текущая задача)

Проект может со временем привлекать несколько ИИ-агентов параллельно (не
только Claude Code как сеньора) — например, отдельный агент для
data-science/BI-части поверх той же кодовой базы. Когда это станет
актуальным, потребуется явно зафиксировать: границы по модулям (кто какие
файлы трогает), общий формат задач (возможно, свой аналог `ANEX-XXX.md` для
другого агента), и то, как изменения одного агента синхронизируются с
другим через git, не через прямое взаимодействие агентов друг с другом.
Не реализуется сейчас — просто зарезервировано место в процессе, чтобы не
изобретать это второпях, когда появится реальный второй агент.
