## Архитектура (кто за что отвечает)

- `db.py` — схема SQLite + вся SQL-аналитика, единая точка правды, от неё
  зависят все модули. `msg_type` по умолчанию `"text"` везде, кроме реакций
  и разбивки по типам — там сознательно все типы сразу. **Фаза 1:** добавлены
  поля `active`/`activated_by` у чатов и авто-миграция схемы — `activity.db`
  больше НЕ нужно удалять при изменении схемы. **AI Activity Feed, Шаг 1:**
  добавляются лёгкие агрегирующие запросы для аномалий флуда, цепочек реплаев,
  коэффициента реактивности и матрицы день×час (см. костыли ниже).
- `crypto.py` — шифрование `text_encrypted` (Fernet), два режима ключа.
  Модуль, который пишет данные, обязан иметь ключ — без него запись
  физически отклоняется.
- `bot.py` — **aiogram, Bot API.** Постоянно живёт в чате, основной механизм
  сбора. Команды `/activate`, `/deactivate`, `/chats`, `/import_history`.
  Выбор чата динамический через команды — фиксированный `TG_CHAT` в `.env`
  больше не используется.
- `history.py` — **точечная догрузка истории личным аккаунтом (Telethon),
  заменяет прежний `collector.py`.** Подключается только на время команды
  `/import_history`, не висит постоянным процессом. Заодно при каждом вызове
  пересчитывает реакции за последние 3 дня (см. костыли ниже).
- `dashboard.py` — Streamlit. Только читает БД, никогда не пишет (кроме
  вызовов внешнего Gemini API из `ai_analyst.py` — см. ниже, это единственное
  исключение из «дашборд ничего не вызывает вовне»). Сайдбар с фильтрами
  (чат/период/тип), вкладки Обзор/Участники/Сообщения/Текст, тепловая карта
  день×час, тренд по дням. **AI Activity Feed:** новая вкладка «AI Инсайды».
- `ai_analyst.py` — **новый (план), интеграция с `google-genai` (Gemini
  API).** Изолированный модуль. Принимает структурированные метаданные из
  `db.py`, отдаёт типизированный JSON для карточек ленты и базовых AI-метрик.
  Не имеет доступа к `crypto.py`/расшифрованному тексту на Шаге 1 — это
  строго Шаг 2.
- `generate_key.py` — утилита генерации ключа шифрования.
- **Хостинг — Fly.io, одна машина на оба процесса (фаза 2, скорректировано).**
  `bot.py` (+ `history.py` по требованию) и `dashboard.py` запускаются внутри
  **одной** Fly Machine через обёрточный скрипт (не как отдельные
  `[processes]` — так они попадают на разные машины с разными volume, см.
  костыль ниже). Один процесс на машину = один физический volume под
  `activity.db` и Telethon session-файл, доступный обоим. Машина —
  `shared-cpu-1x:512MB` (не дефолтные 256MB, см. костыль про память ниже).
  **Имя приложения — `anex-analyst-bot` (ANEX-006, перенос завершён)** —
  старое `tg-chat-analytics` в костылях ниже историческое.

## Костыли, которые не стоит переизобретать заново

- **Privacy Mode в BotFather обязан быть выключен.** По умолчанию Bot API отдаёт
  боту только команды и ответы/упоминания — обычные сообщения чата не приходят
  вообще. Нужно `/setprivacy → Disable` в BotFather **и передобавить бота в чат**
  (смена применяется только при повторном входе). Симптом, если забыли: бот
  «работает», ошибок нет, но в БД почти пусто. Это первое, что проверять при
  жалобе «мало данных».

- **Сбор идёт только для активированных чатов.** Бот получает апдейты из всех
  чатов, где состоит. Хендлер сообщений обязан начинаться с проверки
  `is_chat_active(chat_id)` и молча выходить, если чат не активирован. Без гейта
  бот собирает сразу после добавления — до `/activate`, то есть до второго
  согласия, что ломает саму модель двойного согласия.

- **Дедупликация двух писателей.** В одну БД пишут `bot.py` (live) и `history.py`
  (догрузка назад), их окна пересекаются. Обязателен `UNIQUE(chat_id, message_id)`
  + `INSERT OR IGNORE`. Без этого двойной счёт разъезжается по всей аналитике и
  замечается не сразу. Вместе с `get_peer_id()` это два обязательных условия
  совместимости источников.

- **Bot API — основной сбор, личный аккаунт — только для истории.** Постоянный
  сбор идёт через бота (`bot.py`) — это паттерн, под который Telegram
  проектировал автоматизацию, без вопросов к безопасности. Личный аккаунт
  (Telethon) подключается разово по `/import_history` и не висит постоянно.
  Bot API принципиально не читает историю до момента добавления бота — это
  платформенная граница, не обходится, для истории и нужен `history.py`.

- **MTProto блокируется у пользователя на уровне провайдера.** VPN и публичные
  MTProxy уже пробовали — не сработало надёжно. Рабочее решение —
  `system_version="4.16.30-vxCUSTOM"` при создании `TelegramClient` в
  `history.py` (найдено в коде коллеги). При проблемах с подключением личного
  аккаунта — начинать с проверки этого параметра, а не городить прокси заново.

- **`chat_id` у Telethon и Bot API разный** для супергрупп/каналов (голый id
  vs `-100`-префикс). Теперь в одну БД пишут ДВА источника — `bot.py`
  (Bot API) и `history.py` (Telethon), поэтому везде обязателен
  `telethon.utils.get_peer_id()`, иначе один чат раздвоится в данных.

- **Шифруется только поле text, не вся БД.** SQLCipher рассматривали и
  отклонили: системная C-библиотека, хрупкая установка на скудном окружении.
  Реально волнует содержание сообщений, не метаданные (кто/когда/тип/реакции)
  — на них строится вся аналитика, шифровать их бессмысленно и сломало бы SQL.

- **Два режима ключа шифрования**: `DB_ENCRYPTION_KEY` (статический, в `.env`)
  или `DB_ENCRYPTION_SALT` (пароль). В терминальных процессах пароль — через
  `getpass`, в `dashboard.py` — через `st.text_input(type="password")`. НЕ
  взаимозаменяемо: `getpass` в Streamlit-процессе не работает.

- **Известное ограничение:** ключ сейчас общий на весь процесс дашборда
  (модульная переменная в `crypto.py`). При переходе к шарингу дашборда по
  ссылке (фаза 3) это надо чинить — иначе пароль одной сессии увидят все.
  **Это же ограничение блокирует Шаг 2 AI Activity Feed** (текстовый контур
  требует расшифрованного текста только после явного ввода пароля в текущей
  сессии) — раньше фазы 3 текстовый контур не включать.

- **`_msg_type`: медиа проверяется раньше text.** У Telethon/Bot API `text`
  отдаёт и обычный текст, и подпись к фото/видео — если проверить text первым,
  фото с подписью ошибочно попадёт в "text". Уже было реальным багом; при
  правках в `bot.py` и `history.py` сохранять порядок проверки.

- **Существующие данные не переклассифицируются задним числом.** Авто-миграция
  (фаза 1) меняет схему без удаления `activity.db`, но уже записанные строки
  не переразмечает. Чтобы пересчитать классификацию — осознанно передобавить
  историю, а не рассчитывать на автопересчёт.

- **`st.bar_chart` сортирует строковые категории алфавитно** (дефолт Vega-Lite
  для nominal-полей, не баг версии). Дни недели идут через `st.altair_chart`
  с явным `sort=WEEKDAYS_RU`. Новые графики с именованными категориями (не
  числа/даты) — сразу проверять сортировку, а не только "не упало".

- **`/deactivate` только останавливает сбор, данные не трогает.** Уже собранное
  остаётся доступным для аналитики. Удаление — отдельной явной командой, когда
  понадобится. Причина: случайный `/deactivate` не должен стирать историю
  необратимо.

- **`/import_history` и `/chats` — только владелец деплоя** (`OWNER_USER_ID`
  в `.env`). `/import_history` задействует личный аккаунт владельца через
  Telethon, поэтому админ стороннего чата не должен иметь возможность её
  инициировать. Проверка по user_id, а не по статусу админа в чате.

- **`/activate` и `/deactivate` — админ чата ИЛИ владелец бота, не только
  админ.** Изначально сделано только «админ чата», но по фидбэку пользователя
  после живого теста выяснилось: владелец должен иметь возможность управлять
  сбором в любом чате, даже не будучи там админом (например, в чужой группе,
  где он рядовой участник). `_can_manage_collection()` в `bot.py` сначала
  проверяет `user_id == OWNER_USER_ID`, и только если нет — статус админа
  через `bot.get_chat_member()`. Не путать с правилом выше: `/import_history`/
  `/chats` — **только** владелец, без варианта «или админ» (использует его
  личный Telethon-аккаунт, админ стороннего чата тут ни при чём).

- **`OWNER_USER_ID` — это Telegram user_id, не `TG_API_ID`.** Реальная путаница
  при первом деплое: оба поля — числа, `TG_API_ID` берётся с my.telegram.org
  (идентификатор приложения для Telethon) и не имеет отношения к личному
  Telegram-аккаунту. Если `/chats`/`/import_history` в личке с ботом молчат
  без единой ошибки в логе — первое, что проверять, это не перепутан ли
  `OWNER_USER_ID` с `TG_API_ID`. Узнать настоящий user_id — например, у
  `@userinfobot`.

- **Авторизация Telethon делается заранее и вручную.** Session-файл логинится
  один раз в консоли (код + 2FA). `/import_history` только пользуется готовой
  сессией. Если сессия невалидна — бот отвечает «нужна ре-авторизация в консоли»
  и завершает задачу. Интерактивный логин через чат не делаем: и небезопасно, и
  Telegram блокирует коды, пересланные в самом Telegram.

- **Импорт истории — фоновая asyncio-задача в процессе бота.** aiogram и Telethon
  оба async, живут в одном цикле. Импорт нельзя выполнять синхронно в хендлере —
  polling встанет на минуты. Прогресс отдаётся в чат по ходу. Импорт идемпотентен
  (как прежний `collector.py`): повторный запуск не плодит дубли.

- **`asyncio.create_task()` без сохранённой ссылки может быть собран сборщиком
  мусора прямо посреди выполнения.** Реально воспроизвелось при живом тесте
  `/import_history`: задача подключилась к Telegram и тут же была уничтожена
  («Task was destroyed but it is pending!»). Event loop держит на задачу только
  слабую ссылку. Фикс в `bot.py` — модульный `_background_tasks: set[asyncio.Task]`,
  задача добавляется в него и удаляется через `task.add_done_callback(...discard)`.
  Любой новый fire-and-forget `asyncio.create_task()` в этом проекте обязан
  так же сохранять сильную ссылку.

- **Пересинхронизация реакций — снимок за окно, а не «что изменилось».**
  `history._resync_recent_reactions()` при каждом `/import_history` пересчитывает
  реакции у **всех** сообщений за последние `REACTION_RESYNC_LOOKBACK_DAYS` (3)
  дня, независимо от того, что реально поменялось с прошлого запуска — Telegram
  не даёт способа узнать именно дельту. Отсюда на живом тесте цифра в ответе
  бота оказалась больше, чем ожидал пользователь (обновились и реакции от
  более ранних тестов в том же окне, не только последнее изменение). Сообщение
  в `bot.py` теперь явно поясняет, что это «снимок на текущий момент» — при
  правках текста ответа эту оговорку не терять.

- **Прозрачность для участников.** При `/activate` бот публикует в чате
  сообщение, что сбор активности начат. Команда `/status` доступна любому
  участнику и показывает, собирается ли чат сейчас. Это не UX-украшение, а
  следствие сквозного приоритета приватности: люди в чате должны иметь способ
  узнать о сборе, не спрашивая админа.

- **Бот выселяют из чата → авто-деактивация.** Обработка `my_chat_member`: если
  бота удалили или он потерял доступ, чат помечается неактивным. Иначе он
  останется «активным» в `/chats` навсегда.

- **Ключ шифрования в боте — только `DB_ENCRYPTION_KEY`.** Бот долгоживущий
  процесс, `getpass` под сервисом на хостинге (фаза 2) не работает — stdin нет.
  Режим `DB_ENCRYPTION_SALT` остаётся для дашборда и консольных запусков, для
  бота он нежизнеспособен. Зафиксировано ещё в фазе 1, чтобы фаза 2 не
  переделывала.

- **Выбор чата динамический, `TG_CHAT` удалён.** Чаты регистрируются через
  `/activate`, список со статусом — `/chats` в личке у владельца. Ничего
  «одного чата на деплой» в конфиге больше нет.

- **Почему Fly.io, а не голый VPS (фаза 2).** У бота два долгоживущих процесса
  (`bot.py` + `dashboard.py`) и два файла, которым нужен персистентный диск
  (`activity.db`, Telethon session) — это отсекает PaaS с эфемерной файловой
  системой. Fly.io даёт persistent volumes и встроенные secrets без
  необходимости администрировать ОС (systemd, файрвол, обновления) — оправдано
  профилем пользователя (аналитик, не системный администратор).

- **`DB_ENCRYPTION_KEY` — только через `fly secrets set`, не в `.env` на
  сервере (фаза 2).** `.env` полностью уходит из продакшена начиная с фазы 2,
  остаётся только для локальной разработки. `crypto.py`/`bot.py` читают ключ
  из окружения одинаково в обоих случаях — источник (secrets vs `.env`) для
  кода не должен быть виден.

- **Volume должен существовать до первого деплоя (фаза 2).** `flyctl volumes
  create` — отдельная команда до `fly deploy`; если забыть, `activity.db`
  создастся на эфемерном диске контейнера и исчезнет при следующем редеплое.
  Это самый вероятный способ один раз всё потерять — проверять при первом
  деплое явно.

- **Session-файл Telethon — на том же volume, что и `activity.db` (фаза 2).**
  Оба должны монтироваться из одного persistent volume, иначе `/import_history`
  на новой машине не увидит уже авторизованную сессию.

- **Volume на Fly.io физически привязан к одной машине — не шарится между
  несколькими (найдено на реальном деплое, фаза 2, важная поправка).**
  Исходный план «`bot` и `dashboard` как два отдельных `[processes]` в
  `fly.toml`» не работает: Fly создаёт под каждый process group отдельную
  машину, и при первом же деплое автоматически завёл **второй, независимый**
  volume для `dashboard` — оба назывались «data», но физически разные диски,
  `dashboard` не видел того, что писал `bot`. Это не баг кода, а неверное
  допущение о платформе, вскрывшееся только на живом деплое. **Исправление:**
  `bot` и `dashboard` запускаются внутри **одной** Fly Machine через
  обёрточный скрипт (не два process group) — тогда volume физически один по
  определению, тот же паттерн, что уже проверен локально (`python bot.py` +
  `streamlit run dashboard.py` одновременно на одной машине с общим диском).
  Лишний авто-созданный volume дашборда — удалить после перехода на одну
  машину, оставить только исходный, где уже лежат `activity.db` и session.

- **Вся машина не может автостопиться, пока жив `bot` (уточнение после
  перехода на одну машину, фаза 2).** Раньше предполагалось, что `dashboard`
  как отдельный HTTP-сервис может автостопиться независимо от `bot` — это
  было верно только в мире «два process group», который не сработал (см.
  костыль выше). Теперь `bot` и `dashboard` — один процесс запуска на одной
  машине, `bot` использует long polling и не должен засыпать никогда,
  значит **вся машина** держится постоянно включённой
  (`auto_stop_machines = false` на уровне единственного process group).
  Экономия на автостопе дашборда больше недоступна — на масштабе этого
  проекта (512MB, минимальная стоимость) это несущественно. Про сами 512MB
  и почему их пришлось поднять с дефолтных 256MB — костыль про память ниже.

- **Обёрточный скрипт запуска обязан быть fail-fast, не тихо терять
  процесс.** Если просто запустить `python bot.py &` и `streamlit run
  dashboard.py &` в скрипте без присмотра, падение одного из них (например,
  `bot.py` из-за необработанного исключения) останется незамеченным — второй
  процесс продолжит работать, health-check Fly (если проверяет только HTTP
  дашборда) будет зелёным, а бот при этом мёртв и ничего не собирает. Скрипт
  должен запускать оба процесса, дожидаться завершения любого из них
  (`wait -n` или аналог) и сразу завершать себя (что уронит контейнер и
  вызовет рестарт машины целиком) — простая связка `trap`/`wait` на bash,
  без тяжёлых зависимостей вроде `supervisord`/`s6-overlay`, оправданных на
  куда большем масштабе, чем этот проект.

- **Дефолтные 256MB памяти на Fly Machine не хватает на bot+dashboard вместе
  (найдено на живом деплое, фаза 2-коррекция).** `bot.py` (aiogram+telethon)
  и `dashboard.py` (streamlit+pandas+altair+pyarrow) в одном контейнере — это
  ~90MB RSS у бота и ~40MB у дашборда сразу после старта, до первого реального
  открытия дашборда пользователем. На `shared-cpu-1x:256MB` `MemFree` падает
  до ~2.7MB, машина не крашится с понятной ошибкой, а тихо виснет — не отвечает
  ни health-check, ни даже SSH (подтверждено через `/proc/meminfo`:
  `Committed_AS` уже превышал `CommitLimit`, swap на Fly VM отсутствует в
  принципе). Фикс — `flyctl scale memory 512`, поднято и проверено. Если в
  будущем добавляется что-то ещё тяжёлое в этот же контейнер (например,
  `ai_analyst.py` с `google-genai`) — сразу закладывать память с запасом, а не
  ждать повторного тихого зависания.

- **Fly не перезапускает машину при чистом `exit 0` без явной restart-политики
  (найдено на живом деплое, фаза 2-коррекция).** Дефолт у Fly Machines —
  `restart.policy = "on-failure"`: код выхода 0 трактуется как «намеренная
  остановка», машина остаётся лежать, пока её не разбудит HTTP-трафик к
  дашборду или ручной `fly machine start`. Но `start.sh` при штатном
  fail-fast (SIGTERM одному из процессов, который тот перехватывает и
  завершается корректно) тоже даёт `exit 0` — и в реальном тесте машина после
  такого падения не поднялась сама. `bot` не может себе позволить оставаться
  мёртвым неопределённое время. Фикс — явный блок в `fly.toml`:
  ```
  [[restart]]
    policy = "always"
  ```
  Перезапускает машину при любом завершении, вне зависимости от кода выхода.
  Подтверждено живым тестом: `kill -TERM` процессу бота → весь контейнер падает
  и сам поднимается за ~1.5 секунды.

- **Унификация имени: `anex_analyst_bot` (папка/GitHub/документация) vs
  `anex-analyst-bot` (Fly-приложение), ANEX-006 — выполнено 2026-07-21.** Бот в
  Telegram уже назывался `anex_analyst_bot` (подчёркивание — Telegram не
  разрешает дефис в username). Fly.io, наоборот, не разрешает подчёркивание в
  имени приложения (оно часть домена `*.fly.dev`) — значит там неизбежно
  `anex-analyst-bot` с дефисом. Это единственное осознанное расхождение после
  унификации, не путать со старой случайной нестыковкой
  `tg_chat_analytics`/`tg-chat-analytics`. Fly не поддерживает переименование
  приложения на месте — пришлось создать новое приложение, перенести volume
  (данные скачаны и загружены заново через `fly ssh sftp`, не перемещены
  напрямую — Fly это не поддерживает) и secrets, остановить бота на старом
  приложении **до** запуска нового (иначе два процесса с одним `BOT_TOKEN` дают
  Telegram `409 Conflict`), и только после живой проверки нового приложения
  удалить старое (`fly apps destroy tg-chat-analytics`). Volume ID
  `vol_v3g7m38z9z8eem14` (упоминается в костыле про volume ниже) — от старого,
  уже удалённого приложения; актуальный ID нового volume —
  `vol_r681epkzn7q9lmj4` на `anex-analyst-bot`.
  **Две находки платформы, вскрывшиеся только на живом переносе:**
  1. Путь `dockerfile` в `deploy/fly.toml` резолвится относительно директории
     самого `fly.toml` (`deploy/`), не относительно корня репозитория. Старое
     значение `dockerfile = "deploy/Dockerfile"` при явном
     `--config deploy/fly.toml` даёт задвоенный путь `deploy/deploy/Dockerfile`
     и падает со сборкой. Исправлено на `dockerfile = "Dockerfile"`
     (относительно самого `fly.toml`). Похоже на изменение поведения
     `flyctl` между версиями — если при будущем `fly deploy` снова вылезет
     «dockerfile not found», сверить путь в `fly.toml` в первую очередь.
  2. **Память машины (`fly scale memory 512`) — атрибут конкретной VM, а не
     `fly.toml`, и не переносится автоматически на новое приложение.** Новая
     машина стартовала с дефолтными 256MB и тихо зависла тем же образом, что
     описан в костыле про OOM ниже (health-check `critical`, SSH и `fly logs`
     не отвечают вообще — не только HTTP). Диагностировать помогло
     `fly machine status`: поле `Memory │ 256` сразу указало на причину.
     Фикс — `fly scale memory 512 -a anex-analyst-bot` после первого деплоя.
     При будущих переносах/новых Fly-приложениях для этого проекта — сразу
     закладывать 512MB, не полагаться на дефолт.

- **AI Activity Feed — два контура безопасности, не смешивать.** Метаданный
  контур (Шаг 1: аномалии флуда, цепочки реплаев, реактивность — всё из
  открытых полей `db.py`) работает всегда и не имеет доступа к
  расшифрованному тексту. Текстовый контур (Шаг 2: анализ тем/контекста)
  требует активной сессии с введённым паролем в `dashboard.py` и **не
  включается раньше фазы 3** (см. известное ограничение с общим ключом выше).
  `ai_analyst.py` на Шаге 1 не импортирует `crypto.py` вообще — это
  архитектурная гарантия, а не просто договорённость.

- **Текст, отправляемый в Gemini на Шаге 2, анонимизируется перед отправкой.**
  Имена/юзернеймы участников заменяются на роль/id (например, `Участник_3`)
  до формирования запроса к `google-genai` — контент утекает во внешний API,
  и это осознанное решение, но привязка контента к конкретной личности наружу
  уходить не должна. Реализуется как обязательный шаг пайплайна в
  `ai_analyst.py`, не опция.

- **Лента новостей и AI-метрики обезличены** — никаких прямых указаний на
  конкретного участника («нулевой пациент», «главный распространитель» и
  т.п.) ни в метаданном, ни в текстовом контуре. Формулировки — «кто-то в
  чате», «участник чата» и т.п. Это касается уже Шага 1 (триггеры вроде
  «Мем-Апокалипсис» считаются по метаданным, но в промпте и рендере карточек
  запрещено называть конкретного пользователя).

- **При сбое или лимите Gemini API дашборд показывает последний закэшированный
  результат, не ошибку и не пустую вкладку.** Вкладка «AI Инсайды» — не
  критичный путь; деградация должна быть тихой. `@st.cache_data(ttl="1h")`
  оборачивает вызов, но при исключении из `ai_analyst.py` нужно явно
  перехватывать и отдавать последнее закэшированное значение (или пустое
  состояние с пояснением при самом первом запуске, когда кэша ещё нет).

- **`GEMINI_API_KEY` — через тот же секрет-менеджмент, что и остальные ключи
  (фаза 2, `fly secrets set`).** Не в `.env` на проде, не в коде.

- **AI-метрики («Градус Иронии», «Индекс Душноты» и т.п.) — недетерминированные,
  на них нельзя полагаться как на точные данные.** В UI явно помечать как
  игровые/интерпретационные, отличая от детерминированных SQL-агрегатов на
  остальных вкладках дашборда — иначе пользователь может принять творческий
  вывод LLM за строгую аналитику.

## Как проверять готовое, а не просто "не упало"

`py_compile` ловит только синтаксис. Для реальной уверенности:
- Прогонять функции `db.py` напрямую на синтетических данных (не только на
  пустой базе) — так нашлись реальные баги с фильтрацией и шифрованием.
- Для `dashboard.py` — поднимать настоящий `streamlit run ... --server.headless
  true --server.port N &`, дожидаться `/_stcore/health`, смотреть лог на
  `Traceback`. Bare `python3 dashboard.py` полезен для логики, но там
  `st.stop()` не останавливает выполнение — ветки после него выполнятся; падение
  именно на этом не реальный баг, а особенность bare-режима.
- Для `ai_analyst.py` — тестировать с замоканным Gemini-клиентом (не дёргать
  реальный платный API в каждом прогоне тестов); отдельно — один живой вызов
  вручную, чтобы проверить реальный формат ответа и обработку сбоя/таймаута.
- Для обёрточного скрипта на Fly (фаза 2) — убить один из двух процессов
  вручную внутри контейнера (`kill`) и убедиться, что весь контейнер падает и
  Fly его перезапускает, а не остаётся наполовину живым.

## Рабочий процесс

- **Архитектор в Claude.ai Projects (`PROJECT_CONTEXT.md`)** — архитектура,
  смена парадигмы, выбор между вариантами. Код здесь не пишем и не тестируем.
- **Сеньор в Claude Code / VS Code (этот `CLAUDE.md`)** — вся реализация:
  правки, тесты, запуск.
- Архитектор сам собирает итоговый `CLAUDE.md` с учётом всех принятых решений
  и присылает готовым — пользователь просто кладёт файл сеньору, не
  редактируя вручную и не рискуя перепутать формулировки.
- Внешние концепты (например, RFC от Gemini по AI-аналитике) разбираются
  архитектором перед тем, как стать TASK для сеньора — архитектурные и
  приватностные развилки решает пользователь, техническая реализация уходит
  сеньору только после этого.
- Находки сеньора о платформенных ограничениях (например, поведение Fly
  volume), не видные заранее и вскрывшиеся только на реальном деплое, —
  нормальная часть процесса, не повод считать план проваленным. Архитектор
  оценивает предложенный сеньором фикс, поправляет `CLAUDE.md` и решает,
  требует ли находка более широкого пересмотра архитектуры или это локальная
  коррекция.

## Пользователь

Аналитик, уверенно читает код и работает в консоли, но не разработчик по
профессии. Даёт конкретную обратную связь по существу (реальные баги от
использования). macOS, VS Code. Окружение начиналось на Python 3.9 — перепроверить
перед тем, как полагаться на более новый синтаксис. Ценит проектный,
последовательный подход: обсудить концепт до кода, не забегать вперёд без явного
согласия. Приватность участников чата и безопасность автоматизации для Telegram —
сквозной приоритет во всех решениях, не разовая просьба.
